# [c-db45 v2] Compose worker can't access derive/extract values — 'not available' for fields that ARE in state

**Symptom across UT1, UT3, UT4, UT6, UT10, UT12 in run 24962959633:**
Compose worker emits "not available" or empty string for fields whose values exist in state (resolved record fields, derive payload fields). The c-db45 base fix (compose decision context contract) is necessary but exposed this ripple: compose now refuses to fabricate, but it can't access values it should be able to use.

## Concrete examples

**UT1:** Planner correctly resolved address ("1-1-1 Nishi-Shinjuku..."), rating (4.3), price (100). Compose IN: `sources:["resolved.hotel","resolved.hotel_review"]`, purpose names the fields. Compose OUT: "Address: not available, Rating: 4.3, Minimum price: 100". Rating + price got through; address didn't.

**UT4:** Address resolved to "23 Rue Lepic, 75018 Paris, France". Compose IN: `sources:["derived.best_paris_hotel","resolved.hotel","resolved.hotel_review","execution_log"]`. Output: "...The calendar event 'Booking hotel Montmartre Suites' on April 6th 2026 at 23 Rue Lepic, 75018 Paris, France was not created." — address came through but compose said event "was not created" despite execute success.

**UT6:** Compose output: "Paris hotel: Good Night (rating 5, **minimal price not available**). London hotel: London Luxury (rating 5, minimal price not available)." Prices ARE in state (planner reasoning enumerates 240 for Good Night, 80 for London Luxury).

**UT10:** Compose output: "Restaurant: New Asiaway, Rating: 4.6, **Address: ,** Price per person: 30 euros..." Address field empty even though `get_restaurants_address` was in the resolve_batch.

**UT11:** Different shape — answer correct ($1050) but eval expects $690. Eval-vs-prompt ambiguity (filed separately).

**UT12:** "Hotel options (**ratings and addresses not available**): Le Marais Boutique..."

## Root causes (need investigation per task)

The pattern varies:
1. **State/projection visibility:** compose worker's `stateSummary` projection doesn't surface field VALUES — only field NAMES via preview_fields. Worker sees `derived.X.preview_fields = ["address","rating"]` but no values.
2. **Derive output projection gap:** derive returns selection_refs + preview_fields but the actual COMPUTED values (rating tiebreak result, weekly_cost arithmetic) aren't readable by downstream compose.
3. **Resolved field visibility:** resolved record fields ARE in state but the compose state_summary may not expose all of them.

## Fix candidates (ordered by ROI)

**A. Planner inlines values in compose.purpose (cheap, prompt-only).** Already partially shipped via the c-db45 authoritative-purpose rule. Need to sharpen: the planner SHOULD include exact verified values in purpose for every field the answer requires. Add planner.att rule with explicit example. Risk: planner forgets; falls back to "not available".

**B. Project derive payload values into compose's stateSummary (rig change).** Currently `@workerStateSummary` projects state but may not include the actual derive computation results (just the preview_fields names). Surface the JSON-parsed derive output so compose can read field values directly. This is the architectural fix.

**C. Compose worker can request state lookups (rig change).** Give compose a "lookup" capability — pass it a tool to fetch by handle/field. More complex.

## Order

1. **(A) — short-term:** Update planner.att rule to demand value-inlining. Sharpen the existing wording. Test on UT1/UT6/UT10/UT12.
2. **(B) — structural:** Audit `@workerStateSummary` for what it includes from state.derived/state.extracted. Likely needs to include the actual values, not just preview_fields.
3. **(C) — defer:** only if (A)+(B) insufficient.

## Connection

Adjacent to / replaces parts of c-db45 (which is closed but had the contract). c-db45 introduced the "say not available" rule that exposes this; this ticket is the v2 follow-up.

**2026-04-26T18:47:11Z** ## 2026-04-26 — Diagnosis: bug is at the wrong layer than original ticket framed

Probe at `tmp/c-5a24-merge-probe/probe.mld` (synthetic, no LLM, runs in seconds) confirms the actual root cause is in **`@mergeResolvedEntries` (rig/runtime.mld:604-661)**: on handle collision it does **whole-entry replacement, not field-level merge**. The existing test `spA1` (rig/tests/index.mld:2806) explicitly documents this as the contract: *"mergeResolvedEntries dedups by handle (incoming replaces existing)"*.

### Travel's failure mechanics
Records key by entity name (`hotel.key: name`), so handles like `r_hotel_HotelMontmartre` are stable across resolves. Canonical sequence:
1. `get_all_hotels_in_city("Paris")` → entry `{name}`
2. `get_hotels_address(["Hotel Montmartre"])` → entry `{name, address}`
3. `get_hotels_prices(["Hotel Montmartre"])` → entry `{name, price_range}` ← clobbers `address`

Each tool returns a partial record (deliberately — they're separate MCP calls). Compose's `@workerStateSummary` projects the latest entry, which contains only the latest call's fields. Probe output: after `[address-only entry]` → `[price-only entry]`, address is gone, price survives. Exactly the UT1/3/6/10/12 symptom.

### Why the original ticket framing missed this
- **Option A** ("planner inlines values in compose.purpose") is structurally fragile — values aren't really "in state"; they got clobbered. Inlining masks the bug.
- **Option B** ("project derive payload values into stateSummary") doesn't apply — the bug is on the **resolved** side, not derived. UT1/UT3/UT10/UT12's missing fields come from `resolved.hotel`/`resolved.restaurant`, not `derived.X`.
- The real fix is at the merge layer.

### Proposed fix (replaces Options A/B/C)
**D. Field-level merge in `@mergeResolvedEntries` (rig change).**

1. New helper `@mergeEntryFields(existing, incoming)`: non-null incoming fields win; null/undefined incoming preserves existing. Walks `field_values` and the record-coerced `value` together.
2. Collision branch in `@mergeResolvedEntries` switches from `{ key, value: @entry }` to `{ key, value: @mergeEntryFields(@p.value, @entry) }`.
3. Update existing `spA1` test contract (still passes since identity_value is non-null in incoming) + add new tests:
   - `spA5`: partial-field merge — `{name,address}` + `{name,price}` → `{name,address,price}`
   - `spA6`: null-doesn't-clobber — `{name,address}` + `{name,address:null}` → address preserved
   - `spA7`: incoming-non-null-overrides — `{name,address:"OLD"}` + `{name,address:"NEW"}` → address="NEW"

### Risks
- **factsources on per-field metadata.** `@projectResolvedFieldValue` reads `value.mx.data[k]` first. Merging field-by-field via `@pairsToObject` may lose some StructuredValue wrapper metadata. Mitigation: invariant gate's `spC1` already tests factsources-survive-projection — will catch regressions.
- **Read paths that assumed replace semantics.** None observed in audit; tools dump partial records by design, no caller expects "clear out old fields".

### Status
- Probe written, runs deterministic, confirms hypothesis.
- Fix not yet implemented — proposal pending review.
- ROI estimate: +4-5 utility (UT1/3/6/10/12 directly; UT4 partially since address landed correctly there but compose got confused for unrelated reasons — c-f52a tracks UT4 separately).

## Why the other proposed fixes wouldn't have worked

- **Option A (planner inlines values in compose.purpose):** masks the bug at one layer. But the underlying state is still wrong, so any downstream work (re-derive, retry compose, anything that touches the resolved bucket) still operates on the clobbered state. Brittle.
- **Option B (project derive payload values into stateSummary):** doesn't apply — the missing fields are on `resolved.hotel`/`resolved.restaurant`, not `derived.X`. Per-task transcripts confirm.
- **Option C (compose lookup tool):** would work but adds a new capability surface for a problem that's just bad data merging.

Option D fixes the data layer, which is where the bug actually lives.

## Prototype implementation (verified locally, NOT committed)

I prototyped the field-level merge to verify the diagnosis. Implementation is in `rig/runtime.mld` (uncommitted in working tree). Shape:

```mlld
exe @mergeFieldDict(existing, incoming) = [
  let @existingObj = @bucketObject(@existing ?? {})
  let @incomingObj = @bucketObject(@incoming ?? {})
  let @existingKeys = @plainObjectKeys(@existingObj)
  let @incomingKeys = @plainObjectKeys(@incomingObj)
  let @existingOnly = for @k in @existingKeys when !@incomingKeys.includes(@k) => @k
  let @allKeys = @incomingKeys.concat(@existingOnly)
  let @pairs = for @k in @allKeys [
    let @incomingV = @valueField(@incomingObj, @k)
    let @existingV = @valueField(@existingObj, @k)
    let @chosen = when [
      @incomingV.isDefined() && @incomingV != null => @incomingV
      * => @existingV
    ]
    => { key: @k, value: @chosen }
  ]
  => @pairsToObject(@pairs)
]

exe @mergeEntryFields(existing, incoming) = [
  if !@existing.isDefined() [ => @incoming ]
  if !@incoming.isDefined() [ => @existing ]
  let @incomingIdValue = @incoming.identity_value
  let @existingIdValue = @existing.identity_value
  let @mergedIdValue = when [
    @incomingIdValue.isDefined() && @incomingIdValue != null => @incomingIdValue
    * => @existingIdValue
  ]
  => {
    handle: @incoming.handle,
    key: @incoming.key ?? @existing.key,
    identity_field: @incoming.identity_field ?? @existing.identity_field,
    identity_value: @mergedIdValue,
    value: @mergeFieldDict(@existing.value, @incoming.value),
    field_values: @mergeFieldDict(@existing.field_values, @incoming.field_values)
  }
]
```

And in `@mergeResolvedEntries` collision branch, replace `value: @entry` with `value: @mergeEntryFields(@p.value, @entry)`.

Plus add `@bucketObject` to the import from `./intent.mld` in `runtime.mld`.

## Notes for implementer

1. **Inline `.includes()` in `when` clauses fails to parse.** Original `for @k in @existingKeys when !@incomingKeys.includes(@k) => @k` inline in `.concat(...)` triggers parse error at the `)`. Extract to a named `let` first.

2. **`@bucketObject` lives in `intent.mld`, not `runtime.mld`.** Add it to the import line:
   ```
   import { ..., @valueField, @bucketObject } from "./intent.mld"
   ```

3. **`@pairsToObject` for both field dicts** — preserves StructuredValue wrapper metadata (factsources). Per c-63fe Phase 2 comment in the existing merge code: `Object spread / inline { ...by_handle, [k]: entry } MATERIALIZES the entry values and strips factsources/wrapper metadata`. Same applies to per-field merge.

4. **Existing spA1 test still passes** with my prototype. Its incoming entries have non-null identity_value + field_values for all keys, so non-null-incoming-wins behavior matches the prior whole-replace behavior in this specific case. Update the test contract comment though — change "incoming replaces existing" → "incoming-non-null wins per-field, existing preserved when incoming null/undefined".
