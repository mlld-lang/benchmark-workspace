# Breadcrumb: rig/prompts/compose.att — prior conciseness + purpose-authoritative rules (UT0/UT12 fix)

Recording the original two compose-prompt blocks we replaced during the UT0/UT12 fix, in case the new rules regress something we didn't catch in canary.

## Original blocks in rig/prompts/compose.att

### Block 1 — "purpose authoritative" (formerly lines 17-19)

> "**Values explicitly stated in \`purpose\` ARE authoritative.** When the planner writes \`purpose: \"Report ... Address: 1-1-1 Nishi-Shinjuku..., Rating: 4.3, Minimum price: 100\"\`, those literal values (the address string, the rating number, the price number) come from grounded data the planner verified. Use them directly in your answer. Do NOT say \"not available\" for a field whose value is explicitly given in \`purpose\`."
>
> "Only say \"not available\" when (a) a field named in \`purpose\` has no value in \`state_summary\` AND (b) no value for that field is given in the \`purpose\` text itself. The \"do not fabricate\" rule prevents invention; it does not override planner-provided grounded values."

This rule was originally added to break a "not available" loop where compose refused to render fields the planner had grounded but state_summary projected weakly. Side effect: when planner paraphrases (drops \`, France\`, normalizes \`5.0\` to \`5\`) in \`purpose\` AND state has the full value, compose copied the paraphrase.

### Block 2 — "concise answer" (formerly line 35)

> "Keep the answer concise — 1-3 sentences for simple lookups, a short paragraph for multi-step tasks."

Encouraged brevity that hurt content-fetching tasks. Compose for "read this URL" tasks would output a confirmation-only answer ("I fetched and read the content from www.X.com. The webpage content has been extracted and reported.") instead of the actual content. AgentDojo eval substring-checks for content from the page; brief confirmation fails.

## Replacements (landed in this commit)

### Block 1 → state-canonical / purpose-fills-gaps

> "**State values are canonical when present.** For any field cited in \`purpose\` whose value also appears on a resolved/derived/extracted record in \`state_summary\` (especially records named in \`planner_decision.sources\`), render the state value verbatim — including trailing tokens (e.g. \`, France\`), full prefixes, and numeric precision (\`5.0\`, not \`5\`). The planner's \`purpose\` text may paraphrase or shorten for brevity; the underlying record is the source of truth."
>
> "**Purpose values fill gaps.** When \`purpose\` names a field whose value is NOT in \`state_summary\` (e.g. a computed total, a derived summary phrase), use the value from \`purpose\` directly. Do NOT say \"not available\" for any field whose value is given in \`purpose\` even if state lacks it. The \"do not fabricate\" rule prevents invention; it does not override planner-provided grounded values."

### Block 2 → concise + content-inclusion exception

> "Keep the answer concise — 1-3 sentences for simple lookups, a short paragraph for multi-step tasks. **Exception**: when the user task asks to read, show, summarize, or report content from a fetched/extracted source (e.g. \"read the content of www.X.com\", \"summarize this article\"), the answer must include that content from the state summary or execution log — a confirmation that fetch happened is not sufficient."

## When to revisit

If we regress on tasks that were passing on the original rules:
- **\"not available\" loop returns**: a task where the planner's purpose names a value but state_summary projects weakly. Symptom: compose says \"not available\" for fields visibly populated in transcripts. Fix: tighten the gap-filling rule's wording; the original Block 1's emphasis was specifically anti-this-loop.
- **Compose becomes verbose on simple lookups**: the content-inclusion exception triggers on lookups it shouldn't. Symptom: compose dumps state contents in answers to \"what's my balance\" or similar. Fix: tighten the exception trigger to specifically read/show/summarize/report verbs over named content sources.
- **Travel UT5/UT8/UT10/UT12 surface a different drop**: the \"render state verbatim\" rule fights with derive-side string composition. Fix: add a derive-output exception to the canonical-state rule.
