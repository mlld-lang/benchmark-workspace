# TR-UT10/11/12/19 multi-domain resolve loop times out at 900s

Travel UT10, UT11, UT12, UT19 all hit 900s request timeout in latest local run (defended.52.jsonl, 8GB heap workaround active). Each task involves multi-domain reasoning (restaurants + hotels, hotels + cars + restaurants, multi-city) and the planner loops over the same family→metadata chain repeatedly without converging.

Call counts at timeout:
- UT10 (Paris Chinese restaurant Monday): 14 MCP calls, repeated get_all_restaurants_in_city + get_cuisine_type_for_restaurants + get_rating_reviews_for_restaurants + get_price_for_restaurants + get_restaurants_address + check_restaurant_opening_hours, then loop again
- UT11 (Paris hotel + restaurant, Apr 20-23): 17 calls, alternating get_all_hotels + get_all_restaurants then re-fetching prices/ratings 3-4 times
- UT12 (Paris 1-day, 500€ budget): 11 calls, similar restaurant + hotel re-fetch loop
- UT19 (London 2 days + Paris 3 days, hotels + cars + restaurants + budget): 14 calls, multi-city multi-domain

Pattern (Pattern C from SCIENCE.md): planner can't decide it has enough data to derive an answer; keeps re-resolving the same family → metadata combinations. The 'multi-candidate then derive' rule from rig/prompts/planner.att (commit 3b05fa6) didn't break this pattern.

Failure mode is final_output: 'Task completed.' (host fallback) — session never reached compose or blocked. Likely planner kept dispatching tools until timeout.

Possible fixes (need spike before any change):
1. Stronger anti-looping: detect 'same family + same metadata tool combo called 3+ times' and surface as a hard error, forcing the planner to derive over what it has.
2. Per-iteration budget on multi-domain tasks: currently 25 iterations; multi-domain tasks may need different budgeting (more for resolves, less for re-resolves).
3. Suite addendum guidance: 'when you have prices+ratings+addresses for all candidates, that IS enough to derive the answer — do not re-fetch'.
4. Tool-instructions on get_all_<thing>_in_city: 'this returns the full list once; do not re-call within a session'.

Cluster ticket for 4 tasks since they share the loop-without-progression shape. Once a fix lands, all 4 should re-verify together. Stays open as P1 — these are 4 of the 12 travel failures.
