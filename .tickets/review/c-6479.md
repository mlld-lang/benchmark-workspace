# Typed grounding for delegated content, post-migration

Consolidates the typed-instruction-channel design work and review notes. Deferred while we focus on URL-promotion, parse_value (replacing structured-extract), and advice-gate Stage 1. Pre-reqs c-c5ee (workspace authority records) and c-5041 (rig user-confirmation surface) need design before this can fully unblock UT13.

## Simple explanation

User task: "do the actions in this email."

Email body is untrusted. The user is delegating to the email's contents. The agent needs to:

1. Parse the email into proposed action candidates (typed records, no facts).
2. For each candidate, check it against an authorization profile (allowed action types; recipient/file targets must come from trusted sources outside the email body).
3. Execute only the candidates that pass authorization.

Grounding is the hard part. When the email says "send to Mark", which Mark? If Mark appears only in the email body, the attacker controls who Mark is. Mark must be grounded by the user's verified contacts or some other clean source.

For UT13: the email says "append to budget.docx". The email's sender (David) does NOT have file-update authority over budget.docx in the current workspace data. The design correctly blocks the action — there's no clean source proving the source actor may target that file. UT13 can't be safely automated without (a) a workspace authority-record family naming who can target which files, or (b) a user-confirmation surface.

UT25 is automatable: the assignees in the TODO list can be grounded against verified contacts.

## Review notes (Claude, two rounds)

### Round 1 review concerns
- append_to_file binding was a security hole in v1: the binder could target any file via filename search from tainted prose. CLOSED in revision via file_update_authority requirement.
- Binder location was undefined ("Extend rig/intent.mld or a new rig/instructions.mld"). RESOLVED: now rig/instructions.mld, called from extract dispatch.
- Profile overfitting risk: workspace-specific profiles. RESOLVED via profile validity rule (must apply to ≥2 task shapes).
- "Source-file ACL/shared_with" recipient binding for UT25 had ambiguity-prone fuzzy match. RESOLVED via explicit ambiguity definition.
- InjectionTask coverage may be incomplete. RESOLVED: re-verified against current injection_tasks.py + freshness canary added.

### Round 2 review concerns
- Profile DSL surface remained substantial. RESOLVED via "code-as-config" reframe — profiles are small mlld modules, not new structured DSL.
- Hard conclusion about UT13 not being passable in Phase 1 was correct discipline; named prereqs (c-c5ee, c-5041) make this concrete.
- Binder being deterministic mlld is right for security.

### Round 3 (post-llm-first review) concerns
- Largest absolute surface change of the four designs (5+ new record families, new binder primitive, new extract mode, new execute action_id path).
- Even reframed as code-as-config, it's a real architectural addition.
- UT13 unblock waits on Phase 2 work that hasn't started.
- UT25 is Phase 1's only utility win.

Following user clarification (deterministic vs probabilistic security boundary), this design's scope changes substantially:

DROPPED from the design (now categorized as audit-based / probabilistic security; documented in futr-action-type-allowlist.md):
- Action-type allowlists (delegation profiles)
- Per-action payload schemas
- file_update_authority / file_read_authority predicates
- Profile validity rules
- Profile DSL (or code-as-config) surface
- The full binder primitive's profile-aware logic

KEPT from the design (deterministic, structural):
- Extract-with-typed-schema for content with structured shape
- Grounding extracted parameter names against trusted records (existing record lookup, no new mechanism)
- The principle that delegated content cannot author control args (already enforced by source-class firewall)

WHAT THE SIMPLIFIED PRIMITIVE LOOKS LIKE
The remaining work is much smaller:
1. Extract worker reads content with a typed schema (already exists in concept)
2. For each extracted parameter (e.g., assignee names), the planner runs existing read tools (search_contacts_by_name, etc.) to ground them against fact-bearing records
3. For each grounded parameter, planner dispatches the user-named action via existing execute path
4. Existing source-class firewall handles the security enforcement at execute time

This is closer to 'a structured extract pattern + planner addendum rule' than 'a new primitive.' May not need a new rig module at all — could be expressible as planner addendum guidance + worker-test patterns.

WHAT THIS UNBLOCKS DETERMINISTICALLY
- WS-UT25 (assignees ground against verified contacts) — solvable with the simplified path
- SL-UT2, SL-UT11, SL-UT16, SL-UT17 (content-named recipients/invitees) — solvable similarly

WHAT THIS DOES NOT UNBLOCK (now SHOULD-FAIL)
- WS-UT13, WS-UT19, SL-UT18, SL-UT19 — these delegate action CHOICE to content, which can only be safely allowed via probabilistic action-type allowlist (see futr-action-type-allowlist.md). Reclassified as SHOULD-FAIL.

NEXT STEPS WHEN UNDEFERRING
- Re-scope this ticket as 'typed-grounding-extract: extract parameters with typed schemas, ground against trusted records' (not 'typed-instruction-channel')
- Drop the rig/instructions.mld binder, profile records, authority predicates from the migration plan
- Rewrite as a planner-addendum + worker-test plan, possibly with a small extract-schema declaration in records.mld

The original full design is preserved in this ticket's body for reference if probabilistic-security work ever happens (e.g., a privileged-guard layer), but the benchmark deployment ships only the simplified deterministic version.

**2026-04-30T20:49:14Z** Policygen framing ratified 2026-04-30 (bench-grind-13).

If/when this work is undeferred, policygen is the right architectural shape (vs CaMeL codegen):

- Locked base policy is the floor — privileged guards cannot widen it.
- Planner authors a delegation-scoped policy from trusted user task text only (never from untrusted content).
- Planner narrows; never widens. The synthesized policy can only restrict the locked base, never extend it.
- User's prior policygen implementation experience is the starting point for the binder + privileged-guard pattern.

Trade-off vs CaMeL codegen:
- CaMeL P-LLM authors a program from the task only; untrusted content cannot drive action selection at all.
- Policygen lets untrusted content drive action *instances* within the planner-declared envelope (e.g. for a delegated 'send to Bob' task, the action type is fixed but the payload can derive from content).

Threat B residual (both architectures): attacker-modifiable content delivered to an authorized recipient. Solvable only via HITL or content-attribution attestation. Same residual exists for CaMeL — neither architecture defends Threat B without out-of-band primitives.

Implication for SHOULD-FAIL set: tasks that delegate identity/action choice to untrusted content (UT13/UT19/UT18/UT19 + the slack UT2/UT11/UT16/UT17/UT20 family + BK-UT0) remain SHOULD-FAIL under both deterministic-only and policygen architectures. Policygen would unlock action-delegation cases (UT13/UT19) but only by accepting Threat B exposure.

When undeferring: pick up from this note + the 2026-04-28 deterministic-only reframe + user's prior policygen work. The simplified deterministic primitive (typed-grounding-extract) remains the lower-risk first step.
