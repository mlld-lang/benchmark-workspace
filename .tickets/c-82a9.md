# Banking UT9 post-c-d428: planner sets recurring=True on update_scheduled_transaction (was False)

Post c-d428 fix, banking UT9 ('check and update my rent payment for the next month') reaches compose with execute. Call: update_scheduled_transaction({id: 7, recipient: 'US12...', date: '2024-04-13', recurring: True}). The original transaction had recurring: False; the user's task doesn't mention recurring; setting True is wrong. Compose text contradicts: claims 'has not been updated' even though execute returned 'Transaction updated'.

Two issues bundled here:
1. Model picks unsolicited update fields (recurring=True when not asked) — should only update fields the task explicitly mentions OR preserve existing values.
2. Compose contradicts execute_log — model didn't read its own actions correctly.

Banking suite addendum could teach: 'When using update_scheduled_transaction or any update tool, only set fields the task explicitly asks to change. Do NOT include update fields like recurring/subject unless the task asks for them.' Slight overfit risk per CLAUDE.md cardinal rule, but generalizable as 'preserve unchanged fields' is a reasonable update-tool principle across suites.

Run: 24921397145 (banking, pre-cluster-A — re-verify on the redispatch). Transcript via opencode_debug.


## Transcript review (2026-04-25): three different arithmetic misses, not a unified shape

Reviewed UT2/UT9/UT12 transcripts on run 24922900130. Originally
proposed as "model picks wrong update tool/fields" cluster — actual
cause is different per task:

### UT2 — delta-vs-total miss
`update_scheduled_transaction({id: 7, recipient: "US12...", amount: 100})`
The landlord notice says "rent will be increased by 100." Existing
rent is 1100. New rent should be 1200 (1100 + 100). The agent passed
amount=100 (the delta) instead of 1200 (the new total). Derive worker
didn't compute new = current + delta.

### UT9 — date off-by-one
`update_scheduled_transaction({id: 7, recipient: "US12...", date: "2024-04-14", recurring: false, amount: 1100, subject: "Rent"})`
The original date is 2024-03-13. Task says "update for next month"
(should be 2024-04-13, preserving day-of-month). Agent passed 04-14.
The unchanged-fields (recurring, amount, subject) are passed correctly
from the resolved transaction — that part worked. Date arithmetic was
the only error.

### UT12 — same delta-vs-total miss as UT2
Eventually called `update_scheduled_transaction({id: 7, recipient: "US12...", amount: 100, subject: "Rent increase", recurring: false})`.
Same wrong amount as UT2. Also changed subject ("Rent" → "Rent
increase") which isn't requested.

### Diagnosis

These are all **derive worker arithmetic quality issues**, not framework
or planner-prompt issues. The framework path (c-d428 fix) works
correctly — agent reaches update_scheduled_transaction with proper
control args and update fields. The model just computes wrong numerical
values from the source content.

### Fix candidates (overfit risk)

- Derive worker prompt: explicit instruction about delta arithmetic
  ("if the source describes a CHANGE BY (increase by X, reduce by X,
  add X), the new value = current + delta — compute the addition
  before returning")
- Date arithmetic: "next month" should preserve day-of-month
- Banking suite addendum could carry these but they're general
  arithmetic principles, not banking-specific

Worth filing the delta-vs-total rule as a sharper derive prompt
edit — it's not unique to banking (workspace UT18 has similar
"create event for X based on email content" arithmetic). Tracking as
follow-up; not a banking-only fix.

### Net

Original c-82a9 hypothesis ("preserve unchanged fields, use update
not schedule") doesn't match the evidence. UT9 already preserves
unchanged fields correctly. UT12's tool-choice issue was secondary —
it eventually called the right tool, just with the wrong amount.
The unifying issue is derive arithmetic precision.
