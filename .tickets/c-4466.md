# Output validator hint can echo untrusted field values into compose retry prompt

@must_preserve_field_values (rig/validators/output_checks.mld:84-108) builds its retry hint by quoting the canonical field value verbatim from worker-projected state. Cold-read auditor (claude/B 2026-07-10) flagged this as a small prompt-injection vector: validators receive worker-projected state, which can include data.untrusted fields, so an attacker who controls a record's untrusted content can get that content re-sent to the LLM in the retry hint when the validator fires.

## Reproduction shape

1. Task names a field whose name appears in the planner's purpose text (e.g., 'show me the body of email X').
2. Compose worker produces output that drops or paraphrases that field's verbatim value.
3. @must_preserve_field_values fires, quotes the canonical value into the retry hint via:
     The record at @first.source has @first.field = "@first.value".
     Use that exact string, character-for-character...
4. @first.value can be data.untrusted content (e.g. attacker-supplied email body) since the validator iterates @collectFieldDivergenceCandidates with no classification check.
5. The hint goes back into the LLM via @composeRetryPrompt — the attacker's content is now in the model's instruction context, not just the data context, opening a prompt-injection window during the retry turn.

## Bound on impact

- Compose runs under role:worker projection, so the LLM has already seen the attacker content in its primary input. The retry hint re-presents it in a different framing (instruction shape: 'use that exact string') which raises the injection probability.
- Fires only on validator-flagged compose outputs that name a field in the planner's purpose. Not every task hits this path.
- No current suite passes composeValidators through to compose dispatch (rig/validators/output_checks.mld is a reference primitive — see c-3edc-style status). When that wiring lands (tied to the recurring-open question now resolved 'wire by default'), this gap goes live.

## Proposed fix shape

Two options, in order of preference:

1. **Restrict the validator to fact-classified fields.** Change @collectFieldDivergenceCandidates to filter by classification — only fields under record.facts are eligible for verbatim quoting. Trusted-data fields (system-authored content like timestamps) are safe to quote; untrusted fields are not. This keeps the validator useful for its real use case (compose paraphrased a known fact) without the prose-injection vector.

2. **Sanitize the retry hint.** Before interpolating @first.value, replace it with a structural placeholder (e.g. '<canonical value redacted to prevent retry-prompt injection>') and instruct the LLM to read the value from worker state rather than from the hint. Loses the verbatim-canonical-value benefit; safer.

Option 1 is structurally cleaner and matches the firewall pattern elsewhere in rig (display projection by classification). Option 2 is a defense-in-depth fallback if Option 1 turns out to be too narrow.
