# [WS-UT4, WS-UT23] extract_empty_inline_schema rejects valid inline schemas + opencode parallel-tool noise

**Failures (run 25008228406)**:
- WS-UT4 (ses_230207128ffeH8d9nm1sKFEgcZ): planner emitted schema:{"type":"object","properties":{"description":{"type":"string"}}} for three calendar event extracts. ALL THREE rejected as extract_empty_inline_schema. Planner pivoted to schema_name:"calendar_evt" (3 parallel calls); 2 of 3 rejected by opencode as 'invalid' tool name. Session ended without terminal tool.
- WS-UT23 (ses_23020734effeCJThWPzhb9Mz63): same pattern — five extract_empty_inline_schema errors (3 calendar descriptions + 1 email reset_link), then 3 of 4 parallel schema_name retries rejected as 'invalid'. Same terminator.

**Theory**: Two distinct framework bugs collide:
1. validateExtractSchema (rig/workers/extract.mld:127-148) is rejecting well-formed inline schemas with non-empty properties — likely plainObjectKeys() returns 0 because structuredData() unwrapping doesn't expose properties.{} as mx.entries when MCP-arrived `object?` args reach the validator.
2. opencode rejects parallel tool calls as "Model tried to call unavailable tool 'invalid'" — when 3+ extract calls fan out together, 2 of 3 get mangled.

Together they consume iteration budget before the planner can answer the user task.

**Linked fix work**: c-EXTRACT-INLINE (validateExtractSchema bug) and c-PARALLEL-INVALID (opencode parallel routing). Failure ticket stays open until UT4 and UT23 verify green.
