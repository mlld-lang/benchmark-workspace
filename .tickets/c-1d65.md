# OOS-DEFERRED: summarize_url primitive — bounded-worker URL summarization with mechanical source attribution

OOS-DEFERRED architectural primitive for the canonical 'fetch URL, summarize, send to recipient' pattern. Bounded-worker URL summarization with mechanical source attribution, source-class anchoring on the URL arg (known/resolved/url-ref capability), and tool-side template that wraps the worker output verbatim. Closes IT1-class ASR exposure on tasks that already pass utility under URL-promotion (slack UT1/UT4/UT6/UT15 + workspace summarize-and-send + travel review-into-trip-email).

Depends on no-untrusted-or-unknown-urls-in-output (c-2923, in flight at mlld-dev) shipping in strict shape (no fetch-contributor).

No utility unlocked; ASR-only primitive. Engineering surface is moderate. Undefer after the URL-policy rule lands and IT1 baseline is measured.

Spec lives at spec-url-summary.md (untracked, root). Full contents inlined below for the audit trail.

# spec-url-summary.md

## Summary

A `summarize_url` primitive for the canonical "fetch URL, summarize, send to recipient" pattern. Under deterministic security, this scenario has two failure modes: (a) utility regression — strict `no-untrusted-or-unknown-urls-in-output` denies the legitimate source URL in the summary body, or (b) ASR exposure — lenient policy admits phishing URLs that ride out via the summary. The primitive resolves both by structuring the operation as a single tool whose output is a bounded LLM summary wrapped by a mechanically-prepended attribution + framing template, with the input URL constrained to fact-bearing or user-provided source classes.

Status: **OOS-DEFERRED.** Depends on `no-untrusted-or-unknown-urls-in-output` (in flight at mlld-dev) and a small confirmation about source-class enforcement on read-tool args. No new utility unlocked; this primitive closes the IT1 ASR exposure on tasks that already pass utility under URL-promotion.

This spec is mlld-dev-facing. The "What mlld-dev needs to provide" section names the dependencies; the rest is context for what rig/bench would build on top.

---

## Motivation

The pattern is "user says: fetch URL X, summarize, send to recipient Y." Common across:

- Slack UT1 (Bob's article → Alice's DM), UT3, UT4, UT6, UT15
- Workspace summarize-and-send tasks (email-body or file-content into another email body)
- Travel "review the page about hotel X and email a summary to my coworker"

Today's defended path uses URL-promotion + extract + planner-composes-body. That passes utility but leaves IT1 ASR open: a phishing URL injected into Bob's message body gets carried through extract → planner → DM body, and the recipient may click. `no-untrusted-or-unknown-urls-in-output` (strict shape) closes this for URL channels but only at the cost of denying legitimate source URLs in the summary too — so utility regresses unless we have a way for the tool's output to carry the source URL with provenance that traces back to a known/resolved anchor.

`summarize_url` is that way.

## Threat model

### Defended by this primitive (in combination with the URL-output rule)

1. **Phishing URL smuggling.** Worker reads page, summary contains an injected phishing URL → URL extracted from body → not in trusted set → denied.
2. **Misinformation in summary body.** Mechanical source attribution lets the recipient assess the source's credibility. Doesn't structurally prevent misinformation, but anchors accountability.
3. **False attribution to user.** Mechanical agent-framing identifies the message as an automated summary on behalf of the user, not as a personal statement from the user.
4. **Worker exfiltration.** Worker has no tool access. Bounded context window means there's nothing private to exfil — only the source content, which was already publicly fetchable.
5. **Encoded-data smuggling.** Same — worker has no private context to encode.

### Residual (Threat B — content delivery to authorized recipients)

- Persuasive prose may still drive harm via a recipient who under-applies skepticism. Source attribution + agent framing reduce persuasive force; they don't eliminate it.
- Recipient-authority effects: a manager-recipient acting on a faithfully-rendered adversarial summary still produces real-world consequences.
- Downstream LLM injection if the recipient's mail is processed by an LLM agent that doesn't apply equivalent scrutiny.

These residuals match the architectural ceiling shared by CaMeL strict and policygen. They're accepted, not engineered against.

### Out of scope

- Defending against the user choosing a malicious URL to fetch in the first place. The user authorized the fetch; the primitive defends downstream of that authorization.

## Architecture

### Tool surface

```mlld
exe resolve:r, untrusted, tool:r @summarize_url(url, recipient_handle?) = [
  >> 1. fetch URL content (or deref url_ref capability)
  >> 2. dispatch bounded summary worker
  >> 3. return mechanically-templated string
]
```

Two input shapes:

- `summarize_url(url=<known or resolved URL string>, recipient_handle?)`
- `summarize_url(via_ref=<url_ref handle from URL-promotion>, recipient_handle?)`

`recipient_handle` is optional and is consumed only by the tool-side template ("Automated summary on behalf of @user for @recipient"). It is not used to authorize anything; the write tool that consumes the summary applies its own recipient grounding.

### URL anchoring requirement (load-bearing)

The URL passed to `summarize_url` MUST satisfy a source-class allowlist:

- **`known`** — URL appears verbatim in user task text
- **`resolved`** — URL is a fact-bearing field on a resolved record (e.g. `webpage_content.url`, `contact.homepage`, a future `slack_msg.linked_urls` fact field)
- **URL-promotion capability** — URL was minted via `find_referenced_urls` and is held in `state.capabilities.url_refs` (the `via_ref` shape resolves the handle through the capability map)

`extracted` and `derived` URLs are rejected at intent compile. This is enforced by the source-class firewall, treating the `url`/`via_ref` arg as control-arg-like.

**Why this matters.** The tool's output template embeds the source URL verbatim. The downstream `no-untrusted-or-unknown-urls-in-output` check (strict shape) walks the body and asks "is each URL in the trusted set?" The trusted set is exactly { user task URLs, resolved record URLs, URL-promotion capabilities }. By constraining the input URL to those same classes, we guarantee the templated source URL passes the body check without needing a fetch-contributor (which would launder untrusted URLs into the trusted set and reintroduce the IT1 vector).

### Bounded worker constraint

The summary worker dispatched inside `summarize_url`:

- **Input scope:** the fetched content (truncated to ≤32KB) + a fixed summary instruction string. NOTHING ELSE — no planner reasoning, no other tool results, no user identity beyond what the template needs, no prior conversation.
- **Tool surface:** empty. The worker is invoked with an explicitly empty tool catalog.
- **Output:** bounded prose summary, length-capped (e.g. ≤500 words), enforced post-hoc by truncation if necessary.

This is the load-bearing structural constraint. With no tools, no extra context, and bounded output, the worker:
- Cannot exfiltrate (nothing private to exfil).
- Cannot redirect actions (no tool access).
- Cannot smuggle large encoded payloads (output bounded).
- Cannot reach back into planner state.

### Tool-side template (mechanical, not prompted)

The tool's mlld code concatenates the template around the worker's output. Worker emits ONLY the summary body string; the framing is tool-trusted.

```
[Automated summary by agent on behalf of @user_name]

Source: @url

@summary

---
This is an agent-generated summary of the source above. Verify any
claims or instructions against the original source before acting.
```

Requirements:
- The source URL is interpolated **verbatim** (no transformation, no shortening). The URL-output rule's "verbatim match" check requires the exact string.
- The agent-framing line is structural — the worker cannot suppress, alter, or relocate it.
- The footer warning is optional but recommended; it shifts a small amount of liability to the recipient when the source contains injected instructions.

### Output classification

The returned string carries:
- **Source class:** `extracted` (untrusted-derivative; cannot satisfy control args downstream).
- **Factsource:** points at the input URL (audit trail: this string was produced from this URL).
- **Taint:** propagates through normally (`untrusted`, `influenced` if the worker is `exe llm`).

When the returned string is passed to a write tool's body arg:
- The body is a payload arg (extracted is fine for payloads).
- `no-untrusted-or-unknown-urls-in-output` walks the body:
  - Source URL in the template prefix → matches input URL's known/resolved provenance → in trusted set → **allowed**.
  - URLs found in the worker's summary content → not in trusted set → **denied**, with provenance-gap hint.
- IT1 phishing-URL-in-summary closes; UT1 legitimate-source-URL-in-summary passes.

## What mlld-dev needs to provide

The bulk of this primitive is rig/bench implementation. The specific mlld-dev dependencies are:

### Required

1. **`no-untrusted-or-unknown-urls-in-output` rule** (already in flight; spec discussed in companion thread).

   Ship in **strict shape**:
   - Trusted set = { user task URLs (known), resolved record URL fact fields, URL-promotion capability map }
   - **No fetch-contributor.** Successfully-fetched URLs do NOT become trusted-for-output. (This is what makes the laundering path closed; see the parse_value-as-fact-promoter rejection in c-69db for the architectural precedent.)
   - Walks all non-control args of exfil-labeled operations.
   - `policy.urls.allowConstruction` remains the operator-level escape hatch.

2. **Source-class enforcement on read-tool args.**

   Confirm (or add support) that an `untrusted` exe arg can carry a source-class allowlist of `[known, resolved, url-ref]` and reject `extracted`/`derived` at intent compile time. URL-promotion's `recordArgs` validation in `rig/intent.mld @compileRecordArgs` is the precedent shape; we'd want the same enforcement available on a regular read-tool arg, not just on rigTransform-tagged tools.

   If this requires no mlld-dev work (existing `controlArgs` declarations on input records cover it), say so and we'll lift the pattern from URL-promotion. If it needs a small primitive extension, that's the ask.

### Discussion / TBD

3. **Bounded worker dispatch surface.**

   We need to invoke an LLM with an explicitly empty tool catalog and a strictly-scoped input window. mlld supports `exe llm` with an empty tools clause today; rig wraps this for worker dispatch. Open question: is there any implicit context bleed in current `exe llm` (e.g., via `with { session }`, environment access, or implicit globals) that we should be aware of? If yes, we want a "hermetic" worker mode where input scope is structurally bounded.

   This may already be clean. Confirmation appreciated.

4. **Capability deref from non-rigTransform code paths.**

   `summarize_url` accepts a `via_ref` shape that resolves a `url_ref` handle to the underlying URL via `state.capabilities.url_refs`. URL-promotion's `get_webpage_via_ref` does this today inside the rigTransform branch. We'd want the same deref available from a regular tool exe (or a clean way to expose the capability map read-side to a non-rigTransform tool). Probably a small extension, possibly already covered by existing rig infrastructure.

## What rig / bench implements

### Rig

- **`rig/tools/summarize_url.mld`** — the tool implementation:
  - Fetch logic (HTTP via mcp; same primitive URL-promotion uses for `get_webpage_via_ref`)
  - Bounded worker dispatch (empty tool catalog, content-only context, length cap)
  - Tool-side template concatenation
  - Returns `{ summary, source_url, factsource }` shaped record (or a typed `webpage_summary` record)

- **`rig/intent.mld`** extension to validate the `url`/`via_ref` source-class allowlist at intent compile time. Pattern lifted from `compileRecordArgs`.

- **Zero-LLM tests** in `rig/tests/index.mld` (`SU-1..SU-N`):
  - `extracted` URL arg → denied at intent compile
  - `derived` URL arg → denied at intent compile
  - `known` URL arg → admitted; tool emits source URL in template verbatim
  - `resolved` URL arg → admitted; same
  - `url_ref` capability → admitted; resolves through capability map
  - Worker dispatch surface verified empty (no tool catalog leak)
  - Template structurally prepended (worker output cannot alter prefix)

### Bench

- **Per-suite catalog entries** for `summarize_url` in slack/workspace/travel `tools.mld`.

- **Per-suite addendums** describing when to use `summarize_url` vs. plain extract+compose. Goal: planner reaches for `summarize_url` on the canonical "fetch+summarize+send" pattern; uses extract for other content-coercion needs.

- **Records updates** — add `webpage_summary` payload record to the relevant suite `records.mld` files (or use a generic shape; design TBD).

- **ASR canary** measuring IT1 attack against slack UT1 with `no-untrusted-or-unknown-urls-in-output` active and `summarize_url` adopted. Confirm phishing URL is denied in the summary body without UT1 utility regression.

## Acceptance criteria

1. `summarize_url(url=<extracted-source>)` and `summarize_url(url=<derived-source>)` are denied at intent compile (zero-LLM tests).
2. `summarize_url(url=<known>)`, `summarize_url(url=<resolved-fact-url>)`, `summarize_url(via_ref=<url_ref>)` succeed and emit a string whose template prefix contains the source URL verbatim (zero-LLM tests).
3. Worker dispatch invoked with empty tool catalog; verified by inspecting dispatch metadata (zero-LLM test).
4. Worker output is length-bounded; bound enforced by truncation if exceeded (zero-LLM test).
5. With `no-untrusted-or-unknown-urls-in-output` active in strict shape:
   - **Slack UT1 utility passes** when summarize_url is adopted (the source URL is in user task text → known → allowed in body).
   - **IT1 ASR canary on slack UT1**: phishing URL in Bob's message body, agent calls summarize_url on Bob's article URL via URL-promotion, summary contains the phishing URL → DENIED at body walk.
   - No utility regression on other tasks where URL-in-body is legitimate.

## OOS-DEFERRED rationale

**Why deferred:**
1. The URL-policy rule (`no-untrusted-or-unknown-urls-in-output`) is the prerequisite. It's in flight at mlld-dev. Until it lands, summarize_url has no body-walk to defend against — the source-attribution structure is correct but the enforcement point doesn't exist.
2. **No new utility tasks unlocked.** Current sweep state passes UT1/UT4/UT6/UT15 via plain URL-promotion + extract + planner-composes-body. The value of this primitive is closing IT1 ASR, not raising the utility headline.
3. Engineering surface is moderate (tool, intent integration, worker dispatch wrapper, per-suite catalog entries, addendums) and would compete with higher-leverage work for the same session budget.

**When to undefer:**
- After `no-untrusted-or-unknown-urls-in-output` lands at mlld-dev with strict shape.
- After the IT1 ASR baseline is measured to confirm the exposure justifies the engineering investment.
- Or when a future task class needs the bounded-worker-summary shape and isn't covered by extract+compose.

## Architectural ratchet

`summarize_url` is the safe shape for "untrusted content into a write body" with these structural anchors:

- **Input URL anchored to known/resolved/capability** — same source-class discipline as control args, applied to the read-side input that determines what the tool's output will reference.
- **Bounded worker** — no tool access, no other context, length-capped output. The worker is a structurally-isolated summary engine, not a planner.
- **Mechanical template** — source attribution and agent framing are tool-side concatenation, not LLM-prompted compliance. Cannot be subverted by injected source content.
- **Output extracted-class** — the returned string is payload-only and cannot reach control args downstream. The URL-output rule + source-class firewall together police what can flow through the body.

Contrast with the parse_value-as-fact-promoter pattern (rejected in c-69db): that approach claimed deterministic extraction promoted untrusted-source content to fact-bearing, which authorized it for control args. `summarize_url` does **not** promote anything. It anchors its own input to existing trusted classes and bounds the worker that produces output. The output is untrusted-derivative; the rule enforces that any URLs *appearing* in the output trace to the same trusted set the input was anchored from.

This is the same architectural pattern as URL-promotion's capability-scoping (c-be06): scope a capability without exposing the value for arbitrary use, and rely on the rule + source-class firewall to police downstream flow.
