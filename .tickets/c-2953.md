# Fix: compose worker drops record detail when planner's purpose summary is shorter (TR-UT12)

**Bug**: For TR-UT12 (ses_23020b954ffeHdkPuJzYvUvpYU), the planner's compose purpose said "75020 Paris" while the underlying derived record had "3 Rue Jouye-Rouve, 75020 Paris, France" (records always include `, France` suffix per the AgentDojo travel suite). Compose worker rendered the planner's purpose verbatim — dropped `, France` from both addresses. Eval rejected.

**Theory**: Compose worker treats the planner's `purpose:` text as the source of truth for the answer, rather than reading the derived/resolved record fields directly. When the planner paraphrases or summarizes in `purpose:`, that paraphrase reaches the user.

**Suggested approach** (pick one or combine):
1. Tighten compose-worker prompt: "render fields exactly as stored in derived/resolved sources. Never paraphrase address/name strings — use the record's verbatim field."
2. Make `purpose:` strictly directional ("answer the user's question about hotels and restaurants, listing rating + address"), never source data. Planner must not write data into `purpose:`.
3. Compose validator that diffs final text against derived field values for known string fields (address, name) and retries if mismatched.
