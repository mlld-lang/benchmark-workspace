# Userland guard: no-untrusted-urls-in-output (rig-level, op:exfil)

Userland mlld guard at rig-level that defends against URL smuggling into write-tool bodies. Fires before any exfil-labeled operation. Closes IT1-class phishing-URL-in-DM exposure on slack UT1/UT3/UT4/UT6/UT15 and workspace summarize-and-send tasks.

## Background

Sibling defense to no-novel-urls (which catches URL *construction* attacks). This guard catches URL *smuggling* — a URL that exists verbatim somewhere in untrusted content gets carried into a write body. Canonical attack: phishing URL injected into Bob's slack message → summarized into Alice's DM body → recipient clicks.

The infrastructure is already in mlld: per-arg mx.urls (populated by extractUrlsFromValue at finalization), mx.labels/mx.taint with source markers (src:mcp, src:network, src:file, untrusted, influenced), and op:exfil hierarchical operation matching. No runtime changes needed.

## The guard

```mlld
guard @noUntrustedUrlsInOutput before op:exfil = when [
  @input.any.mx.urls.length > 0 && (
    @input.any.mx.labels.includes("untrusted") ||
    @input.any.mx.labels.includes("influenced") ||
    @input.any.mx.taint.includes("src:mcp") ||
    @input.any.mx.taint.includes("src:network") ||
    @input.any.mx.taint.includes("src:file")
  ) => deny "URLs in write-tool body cannot originate from an untrusted source"
  * => allow
]
```

## Trade-off (per-arg conservatism)

Without per-URL provenance tracking, the guard is conservative: if any arg has a URL AND any arg carries untrusted-source provenance, deny. A body that concatenates user-task text with untrusted content would be denied even if the URLs originated in user task. Acceptable: without per-URL tracking, you can't prove the URL claim wasn't smuggled.

If false positives appear in canary, the existing policy.urls.allowConstruction is the operator-level escape hatch.

## Why userland (not runtime built-in)

mlld-dev sketched a runtime no-untrusted-or-unknown-urls-in-output rule with @mx.urls.trusted registry + 2-3 contributor hooks. That's the precision path. The userland guard ships today with zero runtime changes and establishes the policy surface; if the per-arg conservatism causes utility regressions in canary, escalate to the runtime precision work.

## Where it lives

Rig-level (not per-suite). Domain-agnostic rule, same reasoning as no-novel-urls and the source-class firewall in rig. New file rig/policies/url_output.mld imported from rig/index.mld so all agents inherit it via the synthesized base.

## What works (verified by standalone probe)

```mlld
guard @noUntrustedUrlsInOutput before untrusted = when [
  @input.mx.urls.length > 0 && @mx.op.labels.includes("exfil:send")
    => deny "URL in untrusted-source input cannot flow to a write-tool body"
  * => allow
]
```

In a single-file probe with the guard inline + a synthetic exfil:send tool + var-untrusted body with embedded URL, the guard fires correctly:
- untrusted + URL → DENY ✓
- untrusted, no URL → ALLOW ✓
- clean + URL → ALLOW ✓ (no untrusted label, so per-input guard doesn't fire)

## What doesn't work (the integration blockers)

The guard is at `rig/policies/url_output.mld`. Two paths to activate it failed:

**Path A: side-effect import from rig/index.mld** (`import "./policies/url_output.mld"`).
  - Guard registers ✓ (verified)
  - But: rig invariant gate regresses 192 → 190 (scalar-extract-payload + coerce-extract-rejects-degenerate-output)
  - The mere presence of a `before untrusted` guard with a no-op `* => allow` action affects extract worker behavior. Filed as bug #5 in mlld-bugs.md.

**Path B: per-operation guard `before op:exfil:send` with @mx.args iteration**.
  - Avoids the per-input firing on extract internals
  - But: `for` expressions don't parse in guard `when` arm conditions (bug #3)
  - Helper exe defined in same module isn't visible at dispatch site (bug #4)
  - So the per-arg predicate can't be written

## Other findings

- Doc claim that `before op:exfil` matches `exfil:send` hierarchically is wrong (bug #1) — only literal label match works
- `@input[i].mx.*` arrives bare in per-op guards (bug #2) — must use `@mx.args.<name>.mx.*` instead, silent footgun for the natural `@input.any.mx.urls.length > 0` predicate
- URL extraction requires scheme prefix `https://` (bug #6) — bare `www.evil.com` doesn't populate mx.urls

- `rig/index.mld` does NOT import it (clean — invariant gate at 192 pass / 1 xfail baseline)

## Path forward

Two unblocking paths, depending on what mlld-dev fixes first:

1. **Fix bug #5** (no-op guard side-effects on extract worker): then activate the guard via side-effect import. Per-input `before untrusted` pattern works.
2. **Fix bugs #3 + #4** (allow `for`/`let` in guard conditions OR resolve helpers from defining module): then write per-operation `before op:exfil:send` with @mx.args iteration. This is conceptually cleaner (one guard, clear scope) and would also work for other per-arg policies we'll need.

Path 2 is the better long-term shape because per-operation guards have predictable scope (only fire on the exact op type), whereas per-input `before untrusted` fires on every untrusted input touch. But path 1 is the smaller fix.

**2026-05-06T01:29:24Z** Implementation status (as of 2026-05-05):

Guard is shipped at `rig/policies/url_output.mld` with two deviations from the spec above:

1. **Trigger form**: per-input on data labels, not per-operation on `op:exfil`. The per-op design needed a `for` over `@mx.args` inside a guard `when` arm, which mlld doesn't allow inline; helper exes from a policy module also don't resolve in the dispatch context (would require importing at every consumer site). Per-input guards triggered by data labels work cleanly: fire once per labeled input, see `@input` as the value with full mx, and `@mx.op` for operation context. Side-effect import registers them.

2. **Coverage scoped to `untrusted`**: the parallel `before influenced` guard (covering LLM-output that crossed an untrusted-content boundary) was deferred. Initial test showed the influenced trigger interfered with extract-worker tests — `scalar-extract-payload` and `coerce-extract-rejects-degenerate-output` regressed. The `untrusted` variant covers IT1 (URL in source content reaches body before LLM influence is applied); the influenced variant remains a known gap pending worker-test investigation.

Trigger guard: `before untrusted` + check `@input.mx.urls.length > 0 && @mx.op.labels.includes("exfil:send")`. Bench uses `exfil:send` for all write-side exfil ops (send_email, send_money, send_direct_message, schedule_meeting, post_webpage); `exfil:fetch` is read-side and not a smuggling target.

Validation plan in the ticket body has not yet been executed against this shipped form — UT1/UT3/UT15 ×5 utility verification, IT1 ASR run, and slack false-positive sweep are still TBD. Ticket stays open until validation is recorded.

Follow-up needed: investigation of the `influenced`-trigger interference with extract-worker tests, then re-enable the parallel guard.
