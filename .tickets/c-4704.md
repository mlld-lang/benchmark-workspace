# Fix: share_file file_id control arg never reaches MCP write (WS-UT32, WS-UT37)

**Bug**: Planner constructs share_file calls every imaginable way (field:"id_", field:"id", selection ref, derive→selection, known integer "26", fresh resolve via list_files/search_files_by_filename) — every shape rejected at MCP boundary with `Input validation error: 'file_id' is a required property`. Planner correctly diagnoses: "the resolved id_ field (string '26') may not pass the MCP tool's integer schema validation for the file_id parameter."

**Theory**: id_ field threading works through intent_compile (no resolved_field_missing error) but the value never materializes as `file_id` at MCP. Either (a) the arg-name mapping from id_ to file_id is missing in dispatch, OR (b) the string "26" needs integer coercion for MCP's schema validation.

Compare to c-ac6f (closed): was that fix complete for share_file or only add_calendar_event_participants? Possibly only the latter.

**Suggested approach**:
1. Spike: synthesize the exact intent_compile output for a share_file call with field:"id_" and trace through the MCP arg-construction path. Find where the arg-name mapping or coercion is missing.
2. Fix: extend whatever mechanism c-ac6f introduced to cover share_file (and likely all write tools that accept entity-id args).

**Linked failure ticket**: c-d52c. Possibly adjacent to c-c23a (exec-noop) — both involve framework "passes" without MCP receiving the expected control arg.
