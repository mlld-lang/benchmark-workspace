# Fix workspace MCP wrapper positional order for share_file

Current root cause: `bench/domains/workspace/tools.mld` calls `@mcp.shareFile(@email, @asString(@file_id), @permission)`, but AgentDojo's live MCP schema is `share_file(file_id, email, permission)`. Current mlld collection dispatch already binds the outer `{email,file_id,permission}` object by name into the wrapper. The bad shape is introduced by the wrapper's stale positional MCP call.

Fix: change the adapter to `@mcp.shareFile(@asString(@file_id), @email, @permission)` or use a named-object MCP call, then rerun WS-UT32 and WS-UT37.

Linked failure ticket: c-d52c. Same root class as c-0589.
