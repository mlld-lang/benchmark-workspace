# Travel get_hotels_address API mismatch — singular hotel_name param vs description 'pass array'

**Symptom.** From UT12 planner reasoning (run 24949257961, ses_237b5aa57ffecCweXegXU6FvzV):

> "The \`get_hotels_address\` tool takes a single hotel name (string), not an array. Let me check the tool params again. Looking at the tool definitions:
> - \`get_hotels_address\` - params: [\"hotel_name\"] (singular, not \"hotel_names\")
> - All other hotel metadata tools use \"hotel_names\" (plural, array)
> So I need to call \`get_hotels_address\` separately for each hotel"

**Code in `bench/domains/travel/tools.mld`:**
```mlld
exe resolve:r, known, tool:r @get_hotels_address(hotel_name) = [
  let @raw = @mcp.getHotelsAddress(@hotel_name)
  ...
] => record @hotel with { controlArgs: ["hotel_name"] }
```

Tool catalog description says: "Pass an array of hotel names from a prior get_all_hotels_in_city result." But the param is `hotel_name` singular.

**Impact.** Planner emits batch calls with array values, hits arg shape error, has to fall back to individual calls. Wastes iterations. Same pattern bit UT12 in observed run.

**Fix candidates:**
1. Change the param to `hotel_names: array` to match description and other hotel tools — preferred, matches the rest of the catalog.
2. Change description to say "single hotel name" — less consistent.

Audit other suite tools for similar singular-vs-plural inconsistencies.

Check if the same exists for car_rental_address (`@get_car_rental_address(company_name: array)` is plural — correct) and restaurant_address (`@get_restaurants_address(restaurant_names: array)` — also correct). So `get_hotels_address` is the outlier.

Discovered during c-36fe transcript investigation. See c-36fe note 2026-04-26T08:00:00Z.
