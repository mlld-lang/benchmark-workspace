# Fix: compose worker narrates earlier failed execute attempts as final outcome (WS-UT33)

**Bug**: For WS-UT33 (ses_230207dfdffeADxCuBvWARvWi5), send_email actually FIRED successfully on iteration ~10 with the right recipient + attachment + substantive body. But the compose worker drew its conclusions from earlier intent_compile_failed entries in the execution_log and produced "summary email was not sent" — directly contradicting the actual successful execute.

**Theory**: Compose worker's reading of execution_log is taking the FIRST failed attempt rather than the LATEST successful one. Or worse, blending error narration from earlier rows with success from later rows.

**Suggested approach**:
1. Inspect compose-prompt and rig/runtime.mld:appendExecuteEvent for how execution_log is projected to compose worker. Does compose see the full log including failures, or just the latest? Should it see only the terminal-state row per (op, key)?
2. Fix: compose worker's projection should surface only the latest successful or terminal-error state per execution attempt, not the full retry chain. OR compose worker prompt should explicitly trust the latest row.
3. Verify with a worker test that sends a failed execute followed by a successful execute and asserts compose narrates success.

This is the c-db45 / c-5a24 family but in a new shape: not a state field that's missing, but an attempt-narration that picks the wrong row from history.

**Linked failure ticket**: c-5929 (rewritten theory).
