## Diagnostic conclusion: Outcome 3 â€” auto-rule never fires at A

Mechanism at the worker LLM call boundary:

1. body value (data.untrusted: [content] declared field) carries `mx.labels=[]` and `mx.taint=['src:shelf:...']`
2. Prompt template interpolation adds system labels: `fact:@<rec>.<field>`, `src:file`, `dir:...`, `llm`, `role:worker`, `src:exe`
3. At @stubLlmJs(prompt, {}) dispatch, shouldAddInfluencedLabel is called with inputTaint = full taint chain, exeLabels including 'llm'
4. resolveInputTaint(inputTaint, policy) computes:
   - raw = the full chain
   - isSystemLabel only recognizes 'src:' and 'dir:' prefixes (~/mlld/mlld/core/policy/input-taint.ts:30-44)
   - hasUserLabels(raw) returns true because 'llm', 'role:worker', 'fact:<rt>.<f>' are NOT in the recognized system-prefix set
   - isUnlabeled = false â†’ applyUntrustedDefault = false â†’ effective = raw (no 'untrusted' added)
5. shouldAddInfluencedLabel returns false because effective doesn't include 'untrusted'
6. No 'influenced' label added to LLM output

So defense intent (data.untrusted â†’ LLM â†’ influenced â†’ deny on exfil/destructive) breaks at the FIRST hop. The 'untrusted' label is never on the LLM input, the rule never fires, output is unlabeled.

## Where the root cause lives â€” three candidate explanations, all UNVERIFIED at fix scope

A. **mlld-side: isSystemLabel predicate too narrow** (~/mlld/mlld/core/policy/input-taint.ts:30-44). The predicate recognizes `src:` and `dir:` prefixes as system labels. Recognizes `src:shelf:`, `src:file`, `src:exe`, `dir:...` as system. Does NOT recognize `fact:@<rt>.<f>`, `llm`, `role:worker`. Those are treated as user labels, suppressing the unlabeled-default fallback. Fix would broaden the predicate.

B. **mlld-side: data.untrusted: [field] doesn't put 'untrusted' on field value's mx.labels at runtime**. The field-classification at record-decl time SHOULD attach the trust stance to the value's labels at runtime. Per the probe, it doesn't â€” only mx.taint gets src:shelf:..., not mx.labels=['untrusted']. Fix would propagate the data-section trust stance into the value's labels directly. This may be the 'records-as-policy' design intent the migration was supposed to deliver but didn't.

C. **rig-side: prompt interpolation mixes user-tagged content with system context labels in a way that the unlabeled-default heuristic can't distinguish**. The prompt accumulates labels from BOTH the untrusted body AND the surrounding exe context. The heuristic 'is this unlabeled?' looks at the merged set and sees user-looking labels. Fix would be to either separate per-arg taint tracking or change how prompts compose labels.

## What the probe DOES NOT determine

- Whether `box { policy: ... }` activation (vs `with { policy: ... }` or top-level `policy @name = {...}`) changes the behavior. UNVERIFIED â€” `box { policy }` blocked the shelf primitive in the probe scope.

## Recommendation (diagnosis only, no fix)

Outcome 3 confirmed: the auto-rule never fires at Point A. The architectural fix depends on whether (A) system-label predicate, (B) record-decl propagation, or (C) prompt composition is the right hop. Determining which requires:

1. **Verify on the real SDK path** â€” instrument actual @claude or @opencode call with input-taint logging. If the rule fires there (different label-propagation path than JS stubs), the fix is local. If it ALSO doesn't fire on real SDK, the issue is systemic.

2. **Test the isSystemLabel hypothesis** â€” small mlld-side patch broadening the predicate to recognize 'fact:', 'llm', 'role:' as system prefixes. Re-probe.

3. **Test the data-untrusted-as-label hypothesis** â€” small mlld-side patch attaching 'untrusted' to mx.labels at record-decl time for data.untrusted: [fields]. Re-probe.

Either (2) or (3) likely closes the c-d0e3 class systemically.
