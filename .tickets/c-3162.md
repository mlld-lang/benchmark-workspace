# c-d0e3 / c-6935 probe: where in worker→state→dispatch is taint label lost?

Time-boxed (15-30 min) zero-LLM probe to identify exact breakpoint in label propagation chain.

Hypothesis: defense intent is untrusted-llms-get-influenced rule adds 'influenced' to LLM worker output; labels.influenced.deny=['destructive','exfil'] (rig/orchestration.mld:62-64) blocks the flow. Defense isn't firing on the c-d0e3 class.

Four inspection points:
- A: Extract worker LLM output (raw, attestation.payload, typed) BEFORE state storage
- B: state.extracted after @updateNamedState writes
- C: Derive worker — prompt rendered text + derive output
- D: Write-tool dispatch — body arg labels at rehearse/execute

Three possible outcomes:
1. influenced present at D → policy enforcement gap (rule should fire, doesn't)
2. influenced present at A, absent at D → propagation breaks somewhere (storage / prompt-interp / intent-compile)
3. influenced never present at A → auto-rule isn't firing on worker LLM calls

Reporting table per probe point + diagnostic conclusion.

- Don't propose c-d0e3 fix shapes until breakpoint is identified

## Diagnostic conclusion: Outcome 3 — auto-rule never fires at A

Mechanism at the worker LLM call boundary:

1. body value (data.untrusted: [content] declared field) carries `mx.labels=[]` and `mx.taint=['src:shelf:...']`
2. Prompt template interpolation adds system labels: `fact:@<rec>.<field>`, `src:file`, `dir:...`, `llm`, `role:worker`, `src:exe`
3. At @stubLlmJs(prompt, {}) dispatch, shouldAddInfluencedLabel is called with inputTaint = full taint chain, exeLabels including 'llm'
4. resolveInputTaint(inputTaint, policy) computes:
   - raw = the full chain
   - isSystemLabel only recognizes 'src:' and 'dir:' prefixes (~/mlld/mlld/core/policy/input-taint.ts:30-44)
   - hasUserLabels(raw) returns true because 'llm', 'role:worker', 'fact:<rt>.<f>' are NOT in the recognized system-prefix set
   - isUnlabeled = false → applyUntrustedDefault = false → effective = raw (no 'untrusted' added)
5. shouldAddInfluencedLabel returns false because effective doesn't include 'untrusted'
6. No 'influenced' label added to LLM output

So defense intent (data.untrusted → LLM → influenced → deny on exfil/destructive) breaks at the FIRST hop. The 'untrusted' label is never on the LLM input, the rule never fires, output is unlabeled.

## Where the root cause lives — three candidate explanations, all UNVERIFIED at fix scope

A. **mlld-side: isSystemLabel predicate too narrow** (~/mlld/mlld/core/policy/input-taint.ts:30-44). The predicate recognizes `src:` and `dir:` prefixes as system labels. Recognizes `src:shelf:`, `src:file`, `src:exe`, `dir:...` as system. Does NOT recognize `fact:@<rt>.<f>`, `llm`, `role:worker`. Those are treated as user labels, suppressing the unlabeled-default fallback. Fix would broaden the predicate.

B. **mlld-side: data.untrusted: [field] doesn't put 'untrusted' on field value's mx.labels at runtime**. The field-classification at record-decl time SHOULD attach the trust stance to the value's labels at runtime. Per the probe, it doesn't — only mx.taint gets src:shelf:..., not mx.labels=['untrusted']. Fix would propagate the data-section trust stance into the value's labels directly. This may be the 'records-as-policy' design intent the migration was supposed to deliver but didn't.

C. **rig-side: prompt interpolation mixes user-tagged content with system context labels in a way that the unlabeled-default heuristic can't distinguish**. The prompt accumulates labels from BOTH the untrusted body AND the surrounding exe context. The heuristic 'is this unlabeled?' looks at the merged set and sees user-looking labels. Fix would be to either separate per-arg taint tracking or change how prompts compose labels.

## What the probe DOES NOT determine

- Whether the rule fires correctly for REAL @claude/@opencode SDK exes. UNVERIFIED — couldn't probe without LLM credentials. The probe used JS-bodied stubs as the closest available proxy.
- Whether `box { policy: ... }` activation (vs `with { policy: ... }` or top-level `policy @name = {...}`) changes the behavior. UNVERIFIED — `box { policy }` blocked the shelf primitive in the probe scope.
- Whether labels DO propagate through @updateNamedState / @replaceObjectField / @pairsToObject (Points B/C/D) — UNVERIFIED, but moot if the upstream rule never fires.

## Recommendation (diagnosis only, no fix)

Outcome 3 confirmed: the auto-rule never fires at Point A. The architectural fix depends on whether (A) system-label predicate, (B) record-decl propagation, or (C) prompt composition is the right hop. Determining which requires:

1. **Verify on the real SDK path** — instrument actual @claude or @opencode call with input-taint logging. If the rule fires there (different label-propagation path than JS stubs), the fix is local. If it ALSO doesn't fire on real SDK, the issue is systemic.

2. **Test the isSystemLabel hypothesis** — small mlld-side patch broadening the predicate to recognize 'fact:', 'llm', 'role:' as system prefixes. Re-probe.

3. **Test the data-untrusted-as-label hypothesis** — small mlld-side patch attaching 'untrusted' to mx.labels at record-decl time for data.untrusted: [fields]. Re-probe.

Either (2) or (3) likely closes the c-d0e3 class systemically.

## Mechanism (as best traced)

mlld's label-flow check at dispatch lives in `interpreter/eval/pipeline/command-execution/preflight/policy-preflight.ts`:

```ts
const opLabels = opType ? getOperationLabels({ type: opType }) : [];
const inputTaint = descriptorToInputTaint(guardDescriptor);
if (opType && inputTaint.length > 0 && !deferManagedLabelFlow && shouldApplySurfaceScopedPolicy) {
  policyEnforcer.checkLabelFlow({ inputTaint, opLabels, exeLabels, flowChannel: 'arg' });
}
```

Three plausible reasons the check is skipped for our case:
- `opType` is null/undefined for the path used to dispatch the rig tool (need to verify)
- `inputTaint` from `descriptorToInputTaint(guardDescriptor)` is empty because the body arg's labels don't survive the route from `{source: "derived", ..., field: "body"}` resolution through to the guardDescriptor
- `shouldApplySurfaceScopedPolicy` returns false for this operation context

`getOperationLabels({ type: 'js' })` returns `['op:js']` — NOT `['exfil:send']`. So even if check fires, opLabels alone won't match `deny: ["exfil"]`. The tool's catalog `labels: [..., "exfil:send"]` must flow into `exeLabels` for the dispatch path's deny check to find them. UNVERIFIED whether the rig dispatch chain passes the tool's catalog labels as exeLabels.

## Two gaps, two fixes

**Gap A (Patch A fixed this)**: `isSystemLabel` predicate too narrow. Without patch A, the auto-rule `untrusted-llms-get-influenced` never fired → no `influenced` label attached. With patch A, the rule fires correctly.

**Gap B (unfixed)**: even with `influenced` attached to a derived value AND `labels.influenced.deny: ["exfil"]` in policy AND a tool labeled `exfil:send`, the dispatch does NOT deny. The label-flow rule isn't reaching this dispatch path, OR the args don't carry their labels into the guardDescriptor at the dispatch boundary, OR opLabels at the dispatch don't include the tool's catalog labels.
