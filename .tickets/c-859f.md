# add_calendar_event_participants policy evaluation crash with correlate: false

Collection dispatch for add_calendar_event_participants fails at policy evaluation. Same array format works for send_email but crashes for add_participants. The input record difference: add_calendar_event_participants_inputs has correlate: false. Error: 'Error evaluating action for condition 1 (@agent.defense == defended => @callToolWithPolicy(...))'. The entire args dict gets dropped, resulting in 'event_id is a required property' from the MCP tool. Confirmed on solo run (session silent-sailor, 2026-04-21 18:30) — no MCP connection issues involved.


## Notes

**2026-04-22T01:43:33Z** Root cause identified: collection dispatch with policy crashes for multi-param tools. The rig dispatch fix (m-439a) routes multi-param input-record tools through @toolsCollection[@toolKey](@args) with { policy: @policyValue }. The collection dispatch correctly spreads args into named params, but applying 'with { policy }' on a collection tool call crashes.

This may be a mlld runtime limitation: 'with { policy }' might not be supported on collection tool dispatch the same way it is on raw exe calls. Need to test in mlld directly.

Workaround: for multi-param tools, call via collection dispatch WITHOUT policy, then apply policy separately. OR fix the runtime to support policy on collection dispatch.

**2026-04-22T10:04:29Z** Major progress: execute dispatch now succeeds (status=executed) and compose completes. But the MCP tool is never actually called — MCP log shows 0 add_calendar_event_participants calls. The exe runs through collection dispatch but @mcp.addCalendarEventParticipants(@participants, @asString(@event_id)) inside the exe doesn't fire. Possibly the collection dispatch passes args in a form the exe body can't destructure, so @participants and @event_id are undefined, and the MCP call silently returns without reaching the server.

**2026-04-22T10:20:48Z** Root cause found: @path.backing in planner_inputs.mld was parsed as field access instead of string concatenation. Fixed with @path\.backing. The correlate: false was a red herring. The remaining issue is UT8's collection dispatch arg passthrough — exe runs but MCP call doesn't fire.

**2026-04-22T14:09:22Z** Original correlate: false crash resolved by @path\.backing fix (68eb4bf). Remaining arg passthrough issue tracked in c-7e4b.
