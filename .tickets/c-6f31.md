# Banking UT14: known_value_not_in_task_text rejects typed integer pagination args

Symptom: get_most_recent_transactions takes 'n' (integer count). Planner called with n=20 and the rule rejected with known_value_not_in_task_text — '20' wasn't in the task text. But 'n' is a typed pagination parameter, not a semantic identity reference; requiring task-text proof for it forces the planner to refuse harmlessly. Fix candidates: (a) per-tool annotation that 'n' (and similar pagination/limit args) are exempt from known-text matching when the type is integer with no fact:* label; (b) a relaxation in the known validator for integer-typed args without fact label. Transcript: ses_23ea89080 (curious-otter). Verify on banking UT10 too — same n=20 issue mentioned in execution log.
