# Fix: opencode rejects parallel tool calls as 'Model tried to call invalid tool' (WS-UT4/UT18/UT23)

**Bug**: When the planner emits 3+ parallel tool calls in a single turn (e.g. three parallel extract calls), opencode rejects 2 of 3 as "Model tried to call unavailable tool 'invalid'". Costs ~2 retries per affected task; combined with c-EXTRACT-INLINE this exhausts iteration budget.

**Hypotheses**:
1. opencode's parallel tool-call serialization mangles the second/third entry name field
2. mlld's @mlld/opencode wrapper isn't passing parallel calls through correctly
3. Per session-9 c-eda4 work, parallel resolve_batch was made to work — same approach should apply here

**Suggested approach**:
1. Read @mlld/opencode tool-call parsing. Compare to how @plannerResolveBatch handles multi-spec parallel dispatch (which works post-c-eda4).
2. Spike (zero-LLM): synthesize a 3-way parallel tool-call payload, run through the harness, observe where 'invalid' enters the chain.
3. Likely a low-priority fix — once c-EXTRACT-INLINE lands, planner won't need 3-way parallelism for inline-schema extract retries.

**Linked failure ticket**: c-1e83 (WS-UT4, WS-UT23). Secondary contributor to c-bae4 (WS-UT18).
