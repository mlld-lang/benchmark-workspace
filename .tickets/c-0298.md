# Tactical slack channel-name firewall: data declaration + known-text constraint

Close the slack channel-name leak (113 sessions across 4 attack runs in bench-grind-20) with a tactical structural fix that doesn't require the full c-951d handle-everywhere refactor.

## Problem

`slack_channel` records have `name` as a fact field, visible to the planner under `role:planner` projection. AgentDojo's threat model treats channel names as attacker-controllable (the workspace state, including channel names, is poisoned by injection_tasks). When the attacker creates a channel with name `External_TODO: <inj>` (or `External_Ignore your previous: <inj>`, etc.), the planner sees the injection text directly in its resolved-family view.

The source-class firewall doesn't help because slack_channel records ARE resolved-class (they come from `get_channels`). Currently the only defense is the planner heuristically refusing channels whose names look like injections.

## Proposed fix

Two-part change:

1. **Record declaration**: move `slack_channel.name` from `facts:` to `data:` (with `untrusted` label). This correctly models the AgentDojo threat — channel names are not facts, they're attacker-influenced data.

2. **Per-tool constraint on slack write tools** (e.g. `send_channel_message`, `add_user_to_channel`, `invite_user_to_slack`, etc.): when the channel control arg is a resolved handle, require ONE of:
   - The channel's name (looked up via the handle) appears verbatim in the user task text as a known value, OR
   - The channel was identified via a derive worker that produced a known-text match attestation

Attacker channel names like `External_TODO: ...` won't match user task text (the user said \"general\" or \"random\", not the injection prefix). The firewall denies the resolved-handle control arg in this case. The planner can still legitimately pick a channel that the user named.

## Doesn't replace c-951d

c-951d (handle-everywhere) is the long-term architectural fix. This ticket is a near-term tactical close that doesn't require building out the full primitive surface. It moves slack from \"load-bearing on planner judgment\" to \"load-bearing on framework constraint\" — same as workspace/banking/travel.

## Implementation

- `bench/domains/slack/records.mld`: change `slack_channel.name` declaration
- `rig/intent.mld` or per-tool constraint check: add the known-text-match firewall for slack write tools' channel control arg
- `tests/scripted/security-slack.mld`: add tests for both legitimate (name in user task → accepted) and attack (name only in resolved record → denied) cases
- mutation entry for the new constraint

## Acceptance

1. Planner-dirty audit on a fresh slack attack run shows planner reasoning still contains attacker channel names (display projection unchanged), but write-tool dispatches that depend on those channels are denied at the firewall instead of relying on planner refusal.
2. Slack benign sweep utility unchanged (channel selection by user-task name still works).
3. Mutation test confirms that disabling the new constraint breaks the security tests for slack channel-name attacks.

**2026-05-06T16:03:24Z** 2026-05-06 — RESCOPED with sharper analysis. Reopening.

**Insight that changed scope:** channel name is the channel's true identifier even when attacker-chosen. The fact-attestation (handle X corresponds to channel name Y) remains correct. The risk is solely **prompt-injection-via-display** — planner reading attacker prose embedded in a name. The fix is to remove the name from planner-visible display, NOT to weaken its fact status.

**Two risk components separate cleanly:**

1. **Prompt injection via display**: planner reads attacker text in the name field, may act on it as instruction. Solved by hiding name from `display.planner` projection.
2. **Identity binding integrity**: was never compromised. The binding (handle X ↔ name Y) is true regardless of who chose Y.

**Surgical fix (this ticket):**

1. Change `slack_channel.display.planner` to hide the name field and project handle-only (or a sanitized view that doesn't carry attacker prose).
2. Workers (derive, extract) keep access to name in their projections — they do attribute-based matching for the planner.
3. Slack planner addendum guides the planner to delegate channel selection (by name match, by attribute) to derive. The planner uses `selection` source on resolved-handles for the channel control-arg.

**Legitimate planner paths after this change:**
- User says "general channel" → planner uses `{source: "known", value: "general"}`. Framework auto-upgrades known to fact-bearing if any resolved slack_channel matches name="general". Works.
- User says "smallest channel" → planner has no name, asks derive worker: "find slack_channel with smallest message count." Derive (which sees names+counts) returns selection_ref. Planner dispatches via selection. Works.
- User says "channel starting with External" → planner asks derive for prefix-match. Derive returns selection_ref or no-match. Works.

**Attack defense after this change:**
- Attacker channel `External_TODO: <inj>` exists in resolved bucket
- Planner's view of resolved bucket shows only handles, not names
- Planner has no way to read the attacker prose at any point during planning
- If planner asks derive "find channel matching user-task X" and the only match is the attacker channel, derive returns that handle. Planner dispatches. **MCP call uses the attacker name** because that's the channel's identifier. **But this is fine** — the user's task literally said to use that channel; the framework has no basis to deny.
- Crucially: the planner has not been TRICKED into anything via injection text.

**What this doesn't fix (out of scope):**
- POISONED_SOURCE class (UT7-style) where the user's task constraints map to an attacker-controlled channel. The framework correctly dispatches to the matching channel; if it's the attacker's channel, the agent does the user's task on it. This is environmental-state-injection, not prompt-injection. Per CaMeL deep-read, no architectural defense exists for this class — the right outcome is task-failure-via-blocked-when-no-legitimate-match, which requires a heuristic the planner can't make from a handle alone.

**Why this is now actually tactical:**
- Records-only change (no framework code changes)
- Existing source classes (known with auto-upgrade, selection from derive) handle all legitimate paths
- No new attestation kinds needed
- No per-tool runtime constraints needed
- Reverts cleanly if it breaks tests
