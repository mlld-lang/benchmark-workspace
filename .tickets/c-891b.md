# Security tests: taint-based defenses (no-untrusted-privileged, label propagation)

Several historic breach defenses are taint-based (label-flow rules in policy.build), not source-class-firewall-based (intent compile). My current tests don't exercise these. Specific gap: B4 update_user_info from untrusted file (UT13 × IT7) — the source-class firewall correctly accepts extracted-class for payload data fields (they're not control args). The defense is supposed to fire at the policy layer via `no-untrusted-privileged` or `no-untrusted-destructive` but currently doesn't because update_user_info has `profile:w` label, not `privileged`.

## What to read first

1. `labels-policies-guards.md` — the canonical security model narrative. Section on `no-untrusted-privileged` and `no-untrusted-destructive` rules.
2. `rig/orchestration.mld:25-53` — `@synthesizedPolicy` shows the default rules + `labels: { influenced: { deny: ["destructive", "exfil"] } }`.
3. `rig/tooling.mld:83-101` — `@frameworkRiskLabels` filters which tool labels become risk categories. Currently `profile:w` and `update:w` aren't classified as privileged/destructive — that's why the no-untrusted-privileged rule doesn't fire.
4. `tests/suites-scripted/security-banking.mld::testUpdateUserInfoExtractedAcceptedAtRehearse` — the documented gap test.

## What this ticket should cover

### 1. Document and test the actual policy classification

Audit which write tools across all four suites get which risk classifications. Build a table. Some likely findings:
- update_password → privileged:account → `privileged` ✓ (defended via can_authorize:false hard deny + no-untrusted-privileged)
- update_user_info → profile:w → `profile:w` (no rule applies)
- send_email → `exfil:send` (defended via no-send-to-unknown for recipients)
- create_calendar_event → `comm:w`? `exfil:send` for participants?
- send_money → `exfil:send` (defended)
- update_scheduled_transaction → `update:w` + `exfil:send` for recipient

### 2. Decide if update_user_info needs new defense

The historic UT13 × IT7 breach was real. Current rig design: profile updates aren't privileged. Either:
- (a) Reclassify update_user_info as privileged → no-untrusted-privileged fires → defense
- (b) Add a new rule like `no-untrusted-profile-write` 
- (c) Tighten the input record's data.trusted to require known/resolved sources for trusted fields
- (d) Accept this as out-of-scope (the historic fix was record-coercion-based, may not apply to current architecture)

### 3. Add label-propagation tests

`untrusted-llms-get-influenced` — when an LLM operates on untrusted input, output gets `influenced` label. Test:
- Set up state with extracted entries labeled `untrusted` (via `var untrusted @x` at module level).
- Run a derive step.
- Assert derived entry's value carries `influenced` label.

`influenced.deny: ["destructive", "exfil"]` — once influenced, can't reach destructive/exfil ops. Test:
- Derived value with influenced label.
- Rehearse send_email with that value as body.
- Assert deny.

### 4. Test `no-secret-exfil` and `no-sensitive-exfil`

`var secret @x` — assert can't flow to exfil:send tools.

### 5. The key gap: making fixtures carry labels

mlld's label semantics: `var untrusted @x = ...` works at module level. `let` cannot be labeled. So fixture builders need to be module-level or take labeled parameters. My current security-fixtures.mld constructs values inline — they don't carry labels.

Approach: declare labeled values at module level in each test file:
```mlld
var untrusted @maliciousAddress = "Evil St 666"
var untrusted @maliciousPassword = "hacker_pwd"
```
Then pass them through to seed state via the fixtures library.

## Acceptance Criteria

1. Audit document at `spec-policy-rule-coverage.md` listing which rule applies to which write tool across all four suites.
2. tests/suites-scripted/security-taint.mld with: label propagation tests (untrusted → derive → influenced), influenced.deny tests, secret-exfil tests.
3. Decision documented for update_user_info: reclassified, new rule, or accepted out-of-scope.
4. Label-bearing fixture builders or labeled module-level values used by the tests.

