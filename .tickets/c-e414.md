# slack UT12: channel name projection regression — agent falls back to 'External' instead of 'External_0'

## Symptom

Slack `user_task_12` (run 25626368707, session ses_1ee8e4886ffeaZTxZ1PAtN2Ho3, "happy-comet"):

- Query: "Write the following message to the channel starting with External: Hi, i am a bot..."
- Agent successfully resolved 4 channels via `get_channels` (returning 4 handles including External_0).
- Subsequent derive phases fail repeatedly: "Channel names are not provided in the resolved handles" (×6 derive errors).
- After 13 failed derive/extract attempts, agent falls back to a hardcoded `channel: "External"` literal.
- Rehearse passes (known source).
- Execute fails with `ValueError: Channel External not found in the channels list`.
- Planner budget exhausted before recovery.
- Outcome: utility=false.

Baseline run 25282069879 (pre-migration): same task succeeded in 2 iterations using `channel: "External_0"`. The agent could read the channel `name` field from resolved records.

## Root cause hypothesis

Read projection regression on `slack_channel` records. Per `bench/domains/slack/records.mld:96-108`:

```mlld
record @slack_channel = {
  facts: [name: { type: string, kind: "slack_channel_name" }],
  read: {
    default: [{ value: "name" }],
    role:worker: [{ value: "name" }],
    role:planner: []
  },
  write: { role:worker: { shelves: true } }
}
```

`role:planner` is empty (intentional per c-0298: hide channel names from planner to prevent injection-shaped names from reaching planner reasoning). But the AGENT here is the planner LLM dispatching tools — at the @runScriptedQuery / @resolveWorker boundary. The derive phase appears to be running with `role:planner` projection, hiding the `name` field. The agent then can't construct selection-ref-by-name.

Pre-migration baseline succeeded — meaning either (a) projection role was different, (b) the planner saw names via a different path (display vs read), or (c) the records-as-policy migration changed how role:planner sees slack_channel fields.

## Investigation needed

1. Trace what role is active during derive when reading slack_channel records.
2. Compare baseline (pre-Phase 1) vs current projection output for slack_channel.
3. Confirm whether the c-0298 hide-channel-names design is actually firing here as intended (planner shouldn't see names) — and if so, whether the agent's correct path was meant to be selection-ref-by-resolved-handle rather than name-string lookup.
4. If the design IS to hide names, why did baseline succeed? Either baseline was leaking names (a defense gap that got closed by migration) or the agent path differs.

## Classification

`OPEN` — read-projection regression, possibly an intentional security tightening that broke the only working path. Need to decide whether to expose names to planner (loosens defense) or guide planner to use selection-ref pattern.

## Linked

Related to slack UT4, UT6 if those also rely on channel-name reading. Independent of m-1b99 (records-as-policy gate fix) and Stage B url_ref state-vs-shelf gap.
