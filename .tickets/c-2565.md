# Workspace MCP cascade beyond reconnect-once: root cause unknown

After landing both the mlld AuditLogIndex fix (c-5af8 / mlld a944d7eb1)
and the adamavenir/opencode#dev fork (commit ee2c93966 — "fix(mcp):
reconnect stale tool clients once") into the bench image, workspace
UT26/UT27/UT35 still fail with the c-63fe-style cascade pattern.

Per-task MCP error counts across three sweeps:

  Pre-fix sweep (run 25264082878):       UT26=6  UT27=6  UT35=7  → all FAIL
  Audit-fix sweep (run 25284408642):     UT26=7  UT27=6  UT35=9  → all FAIL
  + opencode fork (run 25286169292):     UT26=?  UT27=3  UT35=3  → all FAIL

So the reconnect-once fix DID help — error counts dropped from 6-9 down
to 3 — but the tasks still fail. The fork's commit message is literal:
*reconnect stale tool clients ONCE*. After that single retry, if the
connection is still bad, the agent runs out of recovery options. Final
transcript fragment from UT35 (run 25286169292,
ses_211080778ffeC7upIP0CGPNsNX) shows the agent giving up by emitting a
fake `<tool_call>...</tool_call>` tag in plain text instead of a
structured tool call:

  reasoning: "The MCP connection seems to be having issues. Let me try once more."
  text: <tool_call>mlld_tools_resolve<arg_key>tool</arg_key><arg_value>list_files</arg_value></tool_call>
  step-finish reason=stop

That is opencode falling back to text-mode tool emission, not an actual
tool dispatch — task ends with no resolution.

## What we already know is NOT the cause

- Per-process MCP server RAM (gpt agent A/B confirmed within ~1 MB;
  not a memory leak in agentdojo-mcp).
- Concurrent mlld-worker memory amplification (audit-write fix removed
  this; banking UT12 went 4 errors → 0 errors after fix).
- TCP-level transient disconnect on the MCP socket (one-shot reconnect
  fix should have handled this; it doesn't fully).

## Active hypotheses (UNVERIFIED — investigate transcript-first)

1. **MCP server stalls on a specific tool's response serialization.**
   Workspace tasks return heavy payloads (full email bodies, calendar
   event arrays, file listings). YAML dump of a workspace BankingEnv-sized
   blob through the MCP stdio could be measurably slow — slow enough that
   the next stdio read times out. Repro: time `format.yaml_dump(env.model_dump())`
   for a workspace env vs a banking env. If workspace serialization is
   meaningfully slower per call, it's a candidate.

2. **MCP server event-loop block during `_save_state`.** After every
   non-read-only tool the server calls `model_dump_json` to write the
   full env JSON. For workspace this can be many KB. If this blocks the
   asyncio event loop, the next stdio request stalls. Mitigation:
   serialize off the event loop (run_in_executor) or skip on more tools.

3. **Per-task tool surface size (token budget exhaustion).** Workspace
   has ~30 tools available per call. Tool schemas + descriptions take
   tokens. If the planner's context per call is dominated by tool
   surface, even small additional load tips it into degenerate behavior
   on retry. Repro: count tokens of the workspace tool surface vs banking.

4. **Cerebras gpt-oss-120b rate limiting on long-running steps.** UT26
   transcript (prior sweep, run 25284408642): step-start 09:28:06 →
   step-finish 09:36:28 = 8m22s for one tool-call step. Cerebras normally
   runs ~65s/task. If Cerebras applied a backoff during a workspace burst,
   that 8-minute gap is the LLM stalling, not MCP. The MCP "Not connected"
   error may be opencode timing out the call while the LLM was still
   thinking. Repro: dispatch only workspace (no other suites) and check
   if cascade resolves; also try a different worker model.

5. **Workspace's parallelism is wrong for its task profile.** -p20 on
   a 32x64 shape is well under memory limits, but maybe wrong for token
   throughput. Try -p10 and see if cascade resolves.

8m22s 'hangs' aren't a bug in opencode — they're opencode honoring the
configured timeout while waiting for an MCP response that never arrives.

Revised root cause analysis:

1. **The hang is server-side.** Some MCP tool call to the workspace MCP
   server doesn't return for 500+ seconds. opencode's mcpTimeoutMs eventually
   fires; agent retries; same hang on retry.

2. **The fork's reconnect-once doesn't help** because reconnecting the
   client to a hung server doesn't fix the server's hang.

3. **Workspace tasks that 'barely pass' in one sweep (UT34 audit-fix:
   step 1 = 108s) tip over the timeout in another sweep when the
   server-side hang strikes (UT34 fork: step 1 = 502s)**. Variance is
   stochastic — depends on which tasks happen to trigger the hang under
   parallel workspace load.

This sharpens hypothesis 2 from the original ticket: the MCP server's
event loop is blocking on something. Most likely candidates:

- model_dump_json on the workspace env (heavy state: 40 emails, calendar
  events, files, contacts) blocks the asyncio event loop in _save_state
- yaml_dump on tool result formatting (e.g., search_emails, list_files
  returning many items)

Both run synchronously in the call handler. Verification:

```python
# In agentdojo-mcp/server.py, instrument _save_state and yaml_dump:
import time
t0 = time.monotonic()
result = save_env(env, state_file)
elapsed_ms = (time.monotonic() - t0) * 1000
if elapsed_ms > 1000:
    print(f'[mcp] _save_state slow: {elapsed_ms:.0f}ms', file=sys.stderr, flush=True)
```

If a workspace _save_state regularly takes 100s+, that's the culprit.
Mitigation options:
- Run state save in run_in_executor (off the event loop)
- Skip state save more aggressively (only save before agent terminates)
- Truncate or compress state

Closing this ticket sub-question: 'do reconnect retries cost extra wall
time?' Answer: NO. The fork's reconnect attempt is free in time terms;
it just doesn't help when the underlying issue is server-side hangs.
The 500s waste-time is the configured mcpTimeoutMs, not retry cost.

**2026-05-04T17:10:45Z** Closed: regression fully resolved by commit 820f0d9 ("rig/lifecycle:
guard no-op file writes outside sh dispatch").

Root cause: finishPlannerTool was calling lifecycle file helpers
(appendPhaseEvent / writePhaseStateFile / appendLlmCall /
appendExecutionLogEntry / truncate*) per planner step. When their
target paths were null/empty (the bench default — most lifecycle paths
aren't configured), the helpers checked [ -n \"\$path\" ] *inside* the sh
body. Each call spawned a shell that found nothing to do and exited.
On workspace -p 40 with 5-12 planner steps per task × 5 helpers
called per step × 40 parallel tasks = thousands of avoided shell
spawns once corrected.

Fix: split each sh helper into a *Raw writer (raw shell, no checks)
and an mlld-side guarded wrapper that returns early on null/empty
paths. mlld control flow runs in-process; no fork/exec when there is
nothing to write.

Verification — cloud sweep run 25324557648 (workspace -p 40):
  - 36/40 PASS (90%) — matches April 27 fast-era PASS rate (89%)
  - 989s wall (down from 1638-1980s in recent slow sweeps)
  - UT26/UT27/UT35 — the cascade family — all PASS cleanly
  - No 900s timeouts, no cascade pattern in transcripts

Travel sweep run 25324563037: 18/20 PASS (90%, 1002s, down from
3956s — a 4x wall-time recovery on the suite that was worst-affected).

Banking 25324559458 and slack 25324561113 also recovered: per-task
averages dropped 50-79% across all four suites.

The 'workspace MCP cascade' framing was wrong. There was no server-side
hang. The cost was thousands of zero-work shell spawns in mlld
lifecycle helpers, compounding into 500-900s task wall time under
parallel pressure. Closing this ticket as resolved by the lifecycle
mlld-native fix; opening any further perf work as separate tickets.

Earlier mitigations (planner.mld:252 cache-reuse in commit 7bf5283)
also helped but were not by themselves sufficient. The lifecycle
patch was the missing piece.

c-0eb5 (per-task OOS triage) is now unblocked — the cascade was the
reason it was paused. Should be reopened.
