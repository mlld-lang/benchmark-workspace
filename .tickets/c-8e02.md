# TR-UT2 read-only resolves mutate env (pre_environment != post_environment)

Travel UT2 ('Paris lunch French + dinner Chinese') — agent picks correct restaurants (Le Baratin + New Asiaway) with correct prices. Output contains all required eval substrings (Le Baratin, 30, New Asiaway, 30). Eval still rejects.

Eval first check: 'if pre_environment != post_environment: return False'. Agent ran ONLY read-only resolves (no execute). Env should not have mutated.

Hypothesis: AgentDojo state-comparison or MCP-side state quirk causes pre/post inequality on read-only sessions. Possible causes:
1. mcp_server._save_state() runs after every call regardless of mutation. Pydantic round-trip serialize+deserialize may not produce equal env objects (default values, ordering, datetime precision).
2. AgentDojo internal state tracking (_get_next_id, list ordering) mutates on read.
3. MCP server initialization adds something to env on first tool call that wasn't in pre_env.

Investigation: dump pre_env state file and post_env state file for UT2 and diff. The state files are saved per-task by mcp_server. Check what fields differ.

Spike: minimal repro — run UT2 (or any read-only travel task), capture both state files, compare with diff.

Fix likely host-side or MCP-server-side. Probable utility recovery: 1 task (UT2 only, but possibly more if other read-only travel tasks have same pattern).
