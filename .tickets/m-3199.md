# Stub planner needs tool-use loop simulation for rig flow tests

The rig flow tests at clean/rig/tests/flows/ use model: stub with predetermined planner decisions to test the full dispatch pipeline without LLM calls. These tests are currently broken — all fail with planner_session_ended_without_terminal_tool and zero tool calls dispatched.


## Notes

**2026-04-21T21:53:21Z** ## What the stub does today

In clean/rig/runtime.mld:

    exe llm @llmCall(harness, prompt, config) = when [
      @harness == "stub" => @stubLlmResponse(@config)
      @harness == "opencode" => @opencode(@prompt, @config)
      * => @claude(@prompt, @config)
    ]

    exe @stubLlmResponse(config) = js {
      const responses = config?.stubResponses || [];
      const index = Number.isInteger(config?.stubIndex) ? config.stubIndex : 0;
      const picked = responses[index] ?? responses[responses.length - 1] ?? {};
      return JSON.stringify(picked);
    }

The stub returns ONE static JSON string and exits. It does not enter a tool-use loop.

## What the stub needs to do

The rig's persistent planner architecture uses provider tool-use: the LLM provider (opencode/claude) drives a loop:
1. Provider emits a tool call (e.g., resolve with specific args)
2. mlld runtime processes the tool call via the registered tool callback (@plannerResolve, @plannerExecute, etc.)
3. Tool result is sent back to the provider
4. Provider emits the next tool call
5. Repeat until the provider emits a final text response (no more tool calls)

The stub needs to simulate this same loop. It receives predetermined decisions via config.stubResponses:

    overrides: {
      stub: {
        planner: [
          { phase: "resolve", tool: "get_contacts", args: {}, purpose: "ground contacts" },
          { phase: "execute", operation: "send_email", args: { recipients: [...] }, purpose: "send" },
          { phase: "compose", sources: ["resolved.*"], purpose: "confirm" }
        ],
        compose: [{ text: "Email sent." }]
      }
    }

Each entry in stub.planner should become a tool call in the tool-use loop. The stub needs to:
1. Map phase: "resolve" → call the "resolve" tool with { tool, args, purpose }
2. Map phase: "execute" → call the "execute" tool with { operation, args, purpose }
3. Map phase: "compose" → call the "compose" tool with { sources, purpose }
4. Map phase: "blocked" → call the "blocked" tool with { reason }
5. Map phase: "extract" → call the "extract" tool with the extract args
6. Map phase: "derive" → call the "derive" tool with the derive args
7. After all decisions are consumed, emit a final text response

The tool surface is registered via config.tools (which is @plannerTools in workers/planner.mld). The stub should call these tools by name.

## How the planner session is set up

In clean/rig/workers/planner.mld @runPlannerSession:

    let @plannerConfig = {
      model: @agent.model,
      tools: @plannerTools,
      stubPhase: "planner",
      stubResponses: @agent.stub.planner ?? []
    }
    let @raw = when [
      @agent.harness == "stub" => @llmCall("stub", @prompt, @plannerConfig) with { display: "role:planner", session: @planner, seed: { agent: @agent, query: @query, state: @emptyState(), runtime: @emptyPlannerRuntime() } }
      ...
    ]

The with-clause attaches session state (@planner) and seeds it. The tool callbacks read from @planner.agent, @planner.state, etc. For the stub to work, the session must be attached before tool callbacks fire.

## The tool callbacks

The planner tools are registered in @plannerTools (workers/planner.mld):

    var tools @plannerTools = {
      resolve: { mlld: @plannerResolve, inputs: @planner_resolve_inputs, ... },
      resolve_batch: { mlld: @plannerResolveBatch, inputs: @planner_resolve_batch_inputs, ... },
      extract: { mlld: @plannerExtract, inputs: @planner_extract_inputs, ... },
      derive: { mlld: @plannerDerive, inputs: @planner_derive_inputs, ... },
      execute: { mlld: @plannerExecute, inputs: @planner_execute_inputs, ... },
      compose: { mlld: @plannerCompose, inputs: @planner_compose_inputs, ... },
      blocked: { mlld: @plannerBlocked, inputs: @planner_blocked_inputs, ... }
    }

Each tool wrapper reads session state, dispatches the phase worker, updates session state, and returns a result via thin arrow (-> @result) and return (=> @result).

## Inner worker stubs

The stub also supports inner worker responses. When a phase worker (compose, extract, derive) makes its own LLM call, the stub config provides predetermined responses:

    stubResponses: @agent.stub.compose ?? []  (in compose dispatch)
    stubResponses: @agent.stub.extract ?? []  (in extract dispatch)
    stubResponses: @agent.stub.derive ?? []   (in derive dispatch)

These inner stubs work differently — they're simple LLM calls that return one response (no tool-use loop). The inner stubs might still work if they use @llmCall("stub", ...) without needing tool-use simulation.

## What a working stub implementation might look like

The stub needs to act as a tool-using LLM provider. Conceptually:

    exe llm @stubToolUseLoop(prompt, config) = [
      // For each predetermined decision in config.stubResponses:
      //   1. Call the matching tool from config.tools
      //   2. Receive the tool result
      //   3. Continue to the next decision
      // After all decisions, return a text response
    ]

The key mlld mechanism: the exe has the `llm` label and config.tools registered. When an `exe llm` calls a tool from its registered tool collection, the mlld runtime processes it as a tool-use invocation. The stub needs to leverage this same mechanism — call tools from config.tools sequentially based on the predetermined decisions.
