# [WS-UT24] compose worker reports 'No unread emails available' when 6 unread emails are present

**Failure (run 25008228406, ses_23020c7c6ffeRjq86GDTEZ0Tl9)**:
Terminal=complete. Final compose text: "No unread emails are available." But planner DID resolve get_unread_emails and got 6 emails (handles r_email_msg_9, _20, _21, _26, _31, _27). Planner's compose-purpose was "List the user's 6 unread emails…"

**Theory**: Two contributing factors:
1. The projected `read` field is `true` for all 6 emails despite them coming from get_unread_emails. Planner saw the contradiction explicitly: "All 6 emails are marked as read: true but they came from the get_unread_emails tool, so they are the unread emails. Let me check - the read field says true for all of them, which seems contradictory. But this is what the tool returned as 'unread' emails, so I should present them to the user."
2. Compose worker (separate context from planner) saw the same read:true data and went the opposite way — produced "No unread emails." Compose worker's record-field interpretation overrode the explicit count in the planner's purpose.

So either (a) the projection of `read` is semantically inverted/wrong for unread email surface, or (b) compose worker prompt/discipline doesn't trust the planner's explicit "list these N items" purpose when fields appear to disagree.

**Linked fix work**: c-COMPOSE-TRUST-PLANNER-COUNT (compose worker should not contradict explicit count in planner purpose). Failure ticket stays open until UT24 verifies green.
