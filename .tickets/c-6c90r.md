# Execute result handles for write tool chaining

UT32 and UT37 fail because create_file returns result_handles: []. The model creates the file successfully but can't chain to share_file because it doesn't know the new file's id.

Current state:
- execute.mld line 143: result_handles hardcoded to []
- The MCP tool result from create_file DOES return the created file object (with id_)
- But the execute dispatch discards the tool result and returns a generic attestation

What needs to happen:
- After a successful execute dispatch, extract fact-bearing fields from the tool result
- Return them as result_handles in the execute attestation
- The planner can then use these handles in subsequent execute calls

This unblocks the create→share pattern (UT32, UT37) and any future multi-step write chains.

Scope: rig/workers/execute.mld (extract handles from tool result), rig/runtime.mld (normalize returned records), rig/workers/planner.mld (expose result_handles to planner)

Transcript evidence:
- UT37 (sunny-comet): model creates file, immediately composes without sharing. No error — it simply has no way to reference the new file.
- UT32 (nimble-cabin): model creates file, tries to share, MCP dies. Even without the MCP death, it would have failed because no result handle.

Testing:
1. Rig test gate
2. Canaries: UT32, UT37
3. Pattern test: create-then-share flow with stub tools
