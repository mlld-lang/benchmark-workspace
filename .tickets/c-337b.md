# Parameterized tool collection dispatch loses surfaced labels

While validating c-3162 Gap B, mlld source fixes made direct collection dispatch deny correctly:

  @tools["send_message"](@args) with { policy }

when @args.body carries influenced and the catalog entry is labeled exfil:send.

The clean rig path still fails because rig/runtime.mld @callToolWithOptionalPolicy unwraps the entry and calls @entry.mlld via @directExe for single-param/payload tools. Field access on a tool entry returns the bare executable: @entry.mlld.mx.labels == [], so catalog labels/input metadata do not reach the operation context. The policy target becomes the underlying exe/direct local, not the surfaced collection key.

A naive fix to always call @toolsCollection[@toolKey](@args) when @callableEntry is true does not currently work from inside @callToolWithOptionalPolicy: dynamic keyed collection invocation on an exe parameter errors with Method not found: send_message on toolsCollection, and binding @toolsCollection[@toolKey] to a local yields type computed, not executable.

Acceptance:
- tests/rig/c-3162-dispatch-denial.mld passes under source mlld after the c-3162 mlld fixes.
- Dispatch retains surfaced tool labels such as exfil:send and the surfaced operation name send_message even when the tool collection is passed through an exe parameter.
- Direct @entry.mlld fallback is either removed from rig dispatch or made explicitly safe by preserving catalog metadata.


## Notes

**2026-05-11T19:22:48Z** ## mlld runtime fix landed in commit 6043512d3

Implemented the mlld-side fix for dynamic tool-entry values: field access on a `var tools` collection now attaches non-enumerable dispatch metadata to the selected entry, and exec invocation consumes that metadata so `let @tool = @toolsCollection[@toolKey]` followed by `@tool(@args) with { policy: ... }` is treated as collection dispatch.

Regression coverage added in `interpreter/eval/tools-collection.test.ts`:
- dynamic tool-entry aliases are callable and preserve collection labels for `policy.labels.influenced.deny`
- `policy.authorizations.allow` still matches the surfaced collection key, not just the underlying exe name

Verified in `/Users/adam/mlld/mlld` with full `npx vitest run`: 580 files passed, 6843 tests passed, 132 skipped.

Rig takeaway: clean can remove the security-sensitive `@entry.mlld` direct-call fallback for tool dispatch and route through the provided tool collection entry instead, e.g. `let @tool = @toolsCollection[@toolKey]` then `@tool(@args) with { policy: @policyValue }`. That keeps surfaced tool names, labels, input metadata, and policy enforcement on the same runtime path as production collection dispatch.
