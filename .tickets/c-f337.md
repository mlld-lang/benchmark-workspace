# Improve no_update_fields error message to be actionable

When execute rejects with no_update_fields, the error says 'Tool X update authorization must specify at least one update field: field1, field2' — this tells the model WHICH fields but not WHY they're missing or HOW to fix it. The model tries different source types (known, derived, selection) and fails every time because the issue isn't sources.

The error should explain: 'The update fields (new_start_time, new_end_time) were not found in the compiled args. Pass them directly as known values or derived refs alongside the control arg. Example: execute({ operation: "reschedule_calendar_event", args: { event_id: <ref>, new_start_time: { source: "known", value: "2026-04-27 10:00" }, new_end_time: { source: "known", value: "2026-04-27 11:00" } } })'

This is part of the broader principle: wrapper error messages are the planner's only teacher.

**2026-04-23T03:54:30Z** The no_update_fields error is now much less likely to fire since the intent compiler includes update fields in the flat policy intent. But if it does fire, the error message should still be improved per this ticket.
