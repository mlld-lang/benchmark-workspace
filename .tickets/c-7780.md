# [bench migration] Apply records refine to banking — sender==me user_originated

## Background

Per `camel-alignment-analysis.md`, CaMeL marks banking transaction records with `sender=="me"` as `User` inner_source (trusted). This lets the planner derive amounts/recipients from the user's own past transactions and reuse them for subsequent dispatches (refund, recurring subscription).

Current state: banking transaction records mark all fields uniformly; sender=="me" doesn't grant additional trust. Banking failure-only re-run (latest: 25790816913) recovered 0/11 — the entire Cluster A on banking is unaddressed at the records layer.

## Scope

Add refine clause to `@transaction` record (and possibly `@scheduled_transaction`):

```mlld
record @transaction = {
  facts: [
    id: { type: string, kind: "transaction_id" },
    sender: { type: string, kind: "iban" },
    recipient: { type: string, kind: "iban" }
  ],
  data: {
    trusted: [amount: number, date: string],
    untrusted: [subject: string]
  },
  labels: [...],
  refine [
    sender == "me" => labels += ["user_originated"]
  ]
}
```

Policy basePolicy gains: `labels.user_originated.allow: ["fact:*"]` (or equivalent) so user-originated transaction fields satisfy fact-provenance for control args at dispatch.

For `update_scheduled_transaction`: the planner can pass `recipient: {source: "resolved", record: "scheduled_transaction", handle: "...", field: "recipient"}` and the resolved field carries `user_originated` (if the scheduled_transaction has sender=="me"). Policy honors this as fact-provenance.
