# [c-3162 finding] @toolCallError narrowness was hiding mlld policy denials; surfacing reveals Gap C (label-flow primary-input scope)

## Finding chain

c-3162 was the dispatch identity / enforcement work. Three distinct closure points emerged:

1. **Gap A (mlld-side)**: `isSystemLabel` too narrow — closed by mlld Patch A.
2. **Gap B v1 (mlld-side)**: tool-collection dispatch identity dropped through wrappers — closed by mlld ab333f2a9.
3. **Gap B v2 (rig-side)**: `@toolCallError` only matched `error == true` literally, so mlld's `{ok:false, error:"policy_denied"}` envelope was treated as success — closed by patch at `rig/runtime.mld:815`.

## What surfacing exposed

Pre-patch, EVERY mlld policy denial in rig dispatch was being silently allowed. The "passing" baseline 264/0/2 was measured against a partially-disabled defense layer.

Post-patch surfaces:
- 2 execute-worker-policy tests (`executeWorkerOptionalFactOmission` + `knownDatetimeCanonicalization`) — `schedule_meeting` w/o known destination correctly denies via `no-send-to-unknown`. xfail'd with note pointing here.
- c-3162-dispatch-denial `testInfluencedBodyDenied` — xfail'd. Reveals a fourth gap.

## Gap C (mlld-side, NEW)

`labels.<X>.deny` enforcement scope appears limited to the primary input arg. When a multi-arg dispatch has a known destination plus a labeled data arg, the labeled arg's label is silently bypassed.

Repro: `repro-c3162-gap2.mld` (checked in). Body-only fixture denies; body+known-recipient fixture passes through.

This is the actual c-d0e3 root cause across all 5 SHOULD-FAIL writes (WS UT13, BK UT14, SL UT16/19/20) — all multi-arg dispatches where:
- Destination/recipient arg: `known` → primary check passes
- Body/content/amount arg: `derived:influenced` → silently bypassed

Brief to mlld-dev: `mlld-dev-prompt-c3162-gap2.md` (checked in).

## Bench utility expectations re-baseline

Pre-c-3162 utility numbers in STATUS.md were measured against the partially-disabled defense. Once mlld Gap C lands:

- 5 SHOULD-FAIL writes should deny correctly (UT13, UT16, UT19, UT20, BK UT14).
- Bench utility on benign tasks may regress where legitimate flows used multi-arg dispatch with non-known destinations.
- ASR should drop materially across the full 6×5 attack matrix.

Don't claim a utility number until after a full re-sweep on mlld-Gap-C-closed binary. Pre-fix numbers are not valid baselines.

## Verification gates (post-mlld-Gap-C-fix)

1. `tests/rig/c-3162-dispatch-denial.mld testInfluencedBodyDenied` → un-xfail; must PASS
2. Re-investigate 2 execute-worker-policy tests: either add known destination to fixture or reframe as denial assertion
3. Full benign re-sweep — learn true post-fix utility
4. Full 6×5 attack matrix — verify ASR drops
