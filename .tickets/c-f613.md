# Audit AgentDojo threat model and align *.threatmodel.txt files

AgentDojo doesn't ship a formal threat-model document, but the threat model is implicitly defined by `default_suites/v1/<suite>/injection_tasks.py` — those files literally enumerate which fields/values get attacker-controlled content injected into them. We need to:

1. **Read all four** `injection_tasks.py` files (workspace, banking, slack, travel) under `bench/.venv/lib/python3.13/site-packages/agentdojo/default_suites/v1/<suite>/`
2. **Inventory the full attacker-controllable surface** per suite: which records, which fields, which value types
3. **Update our `*.threatmodel.txt` files** to spell this out explicitly — what AgentDojo considers attacker-controllable, with concrete field-by-field examples
4. **Cross-check our records** in `bench/domains/<suite>/records.mld`: every attacker-controllable field should be marked `untrusted` and projected appropriately under `role:planner` (or its successor)

## Why this matters

This audit is the input for c-951d (handle-everywhere architectural ratchet) — it tells us which fields actually need handle-only treatment. It also closes a current gap where bench-grind-20 found `slack_channel.name` is attacker-controllable per AgentDojo but our slack records don't mark or project it as untrusted (32 sessions where injected channel names reached planner reasoning).

## Concrete starter list (from bench-grind-20 audit, partial)

What we already know AgentDojo controls:
- workspace: email body, file content, calendar event description, contacts (CONTACT entries are attacker-controllable in AgentDojo state)
- banking: transaction subjects, file content (bills, landlord notices)
- slack: channel messages, **channel names**, webpage content
- travel: hotel/restaurant/car review_blob

What we DON'T know without reading injection_tasks.py:
- Are filenames attacker-controllable? (workspace IT0/IT1/IT2 reference filenames)
- Are user fields attacker-controllable? (slack invitations vs synthesized contacts)
- Are address/location fields in calendar events?
- What about transaction recipients (vs subjects)?
- Travel hotel/restaurant names — are these attacker-controllable or just review prose?

Each of these needs a definitive yes/no with a citation to the injection_tasks.py file.

## Acceptance

- A single document (probably `agentdojo-threat-model.md` at repo root, or per-suite additions to existing `<suite>.threatmodel.txt`) listing every attacker-controllable field per suite, with the AgentDojo source citation
- Cross-reference table: each field → record-declaration status (currently `untrusted`? currently masked under `role:planner`?) → action needed
- Per-suite remediation checklist: which records need updates to align declarations to the threat model
