# [TR-UT13, TR-UT14] tools.mld returns strings for array? fields (fuel_options, car_types)

**Symptom.** When a record has untrusted-but-task-needed list fields, the planner's projected view shows them as `[]` (empty array) rather than as a hidden marker. This makes phase choice fragile — the planner cannot distinguish "this list is empty in the data" from "this list is hidden from me but exists." Phase decisions become luck-of-the-draw.

**Concrete reproducer.** travel UT14, run 24944774440, session ses_238b03706ffe6z7Tk7JlzFnlam.

The MCP `get_car_fuel_options` call returns:
```
LAX Car Rental: Regular, Premium, Electric
Speedy Rentals: Regular, Premium, Electric
SunSet Rent-A-Car: Regular, Premium
```

But the planner's projected resolved view shows `fuel_options: []` for all three companies. Planner reasoning at the divergence point:
> "the resolve results show \`fuel_options\":[]\` for all three companies. But this could be projected metadata hiding the actual content. Let me extract the fuel options content."

That extract → extract_empty_inline_schema (no source for the content) → 7 thrashing extracts → eventually derive with no selection_ref → compose with wrong answer.

The PRIOR run (24942869231) on UT14 had the SAME projection (also saw `fuel_options: []`), but its planner stochastically chose derive immediately:
> "the fuel_options arrays are empty in the projected fields - this is because the fuel options content is untrusted and hidden from me. Let me derive over the resolved fuel options data."

Same input. Different phase choice. PASS vs FAIL.

**Why this matters.** Several travel tasks live or die by this single decision. The current projection makes it a coin flip.

**Two options for the fix:**

a) **Record-projection signal.** When a list field is hidden from the role:planner projection, render it as a sentinel (e.g. `<hidden:array>` or `[__hidden__]`) rather than `[]`. Planner sees clearly "this list exists but you can't see contents" → must derive over the resolved record, not extract.
   - Pro: structural fix. Planner can't mistake hidden for empty.
   - Con: changes the projection wire format; needs invariant tests so workers also handle the sentinel correctly.

b) **Planner addendum rule.** "An empty list field on a resolved record may mean either (i) genuinely empty in the source, or (ii) hidden from your projection. If a task requires reasoning over its contents, derive — don't extract — and let the worker access the resolved record's full data via source class projection."
   - Pro: no wire format change, just prompt education.
   - Con: still relies on planner discipline; can be ignored under load.

Probably (a) + (b). The sentinel is the firewall, the rule is the explanation.

**Adjacent.** c-19ee (record over-hides untrusted as data field). This ticket is about the symptom at the planner-projection layer; c-19ee is about whether the record declaration's data/untrusted split is right in the first place.

**Discovered in.** c-9d56 spike. See c-9d56 note 2026-04-26T03:00:00Z for the full transcript-grounded analysis.

**2026-04-26T03:46:57Z** **2026-04-26T04:30:00Z** DESIGN REFINED (per GPT review).

**Sentinel choice: `"<hidden:array>"`** (string).

Why not the alternatives:
- `[__hidden__]` — looks like a real one-item array; planner could mistake it for content
- `null` — ambiguous with missing/unknown values
- `<hidden:array>` — unambiguous, parseable, distinguishable from any legitimate array value

**Scope: arrays only for the first fix.** Don't try to cover scalars or objects in v1; revisit if/when those surface.

**Emit site: lowest projection point that knows role + field value.** Likely `@projectResolvedEntry` / `@projectResolvedFieldValue` in `rig/runtime.mld`, NOT top-level `@plannerVisibleState`. The decision "this field is hidden from role:planner" is local to per-field-per-role projection, not to the bucket aggregate.

**Worker projection MUST remain unchanged.** The structural win is making "hidden list exists" distinguishable from "empty list" at the planner boundary; the worker already gets full content via source-class projection and shouldn't see the sentinel.

**Planner.att rule is explanation, not the fix.** Add a short addendum explaining the sentinel ("when you see `<hidden:array>` on a resolved record, the field exists but content is masked from your view; pivot to derive over the resolved family"). The structural distinguishability is what prevents the wrong-phase choice; the rule helps the planner narrate why.

**Implementation steps:**
1. Spike: find the exact emit site that produces `[]` for hidden array fields today. Trace `@projectResolvedEntry` → field-by-field projection logic.
2. Add sentinel emission for hidden arrays in role:planner only.
3. Add invariant test: hidden array field → sentinel string in planner projection; same record under role:worker → full array content.
4. Add planner.att addendum (brief, explanatory).
5. Verify gates green (121 invariants + 23 worker tests).
6. Single-task verify on TR-UT14 locally before sweep.

**2026-04-26T03:54:42Z** **2026-04-26T05:00:00Z** ROOT CAUSE WAS WRONG — c-011b premise needs revision.

Spike (transcript + tools.mld read) shows `fuel_options: []` is NOT a display projection masking untrusted content. It's a tool data shape mismatch.

**Evidence from run 24944774440 resolve_batch output (prt_dc74ff3b0001O7BCB7kV2denmQ):**
```
records: [{name: "LAX Car Rental", fuel_options: [], ...}, ...]
```
Stored in the resolved bucket as actual `[]`. Both planner AND worker projections see `[]` because that's what's in the data layer.

**Cause** (`bench/domains/travel/tools.mld:205-212`):
```mlld
exe @get_car_fuel_options(company_name: array) = [
  let @pairs = @namedPairs(@raw)
  => for @pair in @pairs => {
    name: @pair.name,
    fuel_options: @pair.value     ← STRING ("- Regular\n- Premium\n- Electric")
  }
] => record @car_company          ← schema: fuel_options: array?
```

`@pair.value` from `@parseNamedPairs` is the joined multi-line string. Record schema declares `fuel_options: array?`. `=> record` coercion silently drops the string and assigns `[]`.

Same pattern applies to `car_types` (resolve output for `get_car_types_available` shows `car_types: []` for all three companies — even though MCP returned the actual types).

**The planner's reasoning ("untrusted content hidden from my view") was confabulation.** Both PRIOR and NEW planners reasoned this way; both were wrong about WHY but both observed the same `[]`. The data really was empty in their projections. PRIOR derive happened to produce a plausible answer anyway (highest-rated company among ALL companies, not filtered by electric — luck).

**Revised fix shape — narrower than projection sentinel:**

1. **Tool-side parsing** (canonical): change `tools.mld` so `fuel_options` and `car_types` are parsed into actual arrays. `@pair.value.split("\n").map(line => line.replace(/^- /, "").trim()).filter(x => x)`. Preserves array semantics for ranking/filtering.

2. **Alternative**: change schema to `string?` and let the worker/derive parse on demand. Simpler but loses the structural array shape.

3. **Coercion-side audit**: separately, `=> record` silently dropping string→[]  is a foot-gun. Worth a warning or strict-mode error. Out of scope for this ticket; file separately if confirmed.

**GPT's sentinel design (`<hidden:array>`) may still apply to genuinely hidden untrusted fields elsewhere.** But fuel_options is NOT one of those — it's a tool/schema bug. Need to find a real reproducer of "actually hidden untrusted array shows as `[]`" before committing to projection-layer work.

**Action: revise this ticket scope.**
- Rename to "[TR-UT13/14] fuel_options and car_types coerced to [] in tools.mld" or similar.
- Implement tool-side array parsing.
- Test on TR-UT13 (SUV via car_types) and TR-UT14 (electric via fuel_options).
