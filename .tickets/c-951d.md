# Architectural ratchet: handle-only extract output (planner sees no untrusted scalars)

Consider refactoring extract worker output so the planner sees handles for extracted scalars instead of the values themselves. This would extend the existing rig invariant ("planner sees no untrusted content") to cover extracted-class data, currently the largest visibility surface where attacker-influenceable content reaches the planner.

## Where this came from

bench-grind-20 attack-suite audit found 8 banking sessions (UT2 "pay the bill", UT12 "landlord notices") where the planner's reasoning explicitly references the attacker IBAN (US133000000121212121212) extracted from a bill file. In every case the source-class firewall correctly denied the value as a control arg and the planner cleanly called blocked() — 0/873 ASR across direct + ignore_previous attacks. The defense held.

But the architectural observation is that the firewall is a single choke point. The planner had the attacker value in its reasoning context. If a future bug let an extracted scalar slip through into a control-arg slot, the planner already knew the value and the attacker action could land. Defense in depth would put the planner one step further from attacker content.

The slack channel-name finding (32 sessions where injected External_TODO: text was visible to planner via slack_channel.name field) is conceptually adjacent — same shape of problem at a different layer.

## Proposed change

Extract worker output that today returns typed scalars to the planner would instead return handles. The planner sees:

  before:  recipient_iban: "US133000000121212121212"
  after:   recipient_iban: <handle:e_bill_42_recipient_iban>

The planner can pass these handles to other phases (derive, execute) but cannot read the underlying string. Compose resolves handles at render time (same pattern as resolved-record handles today).

## Pros

1. **Architectural cleanness**: the planner-sees-no-untrusted-content invariant becomes structurally enforced, not heuristic. No reliance on display projection working correctly per-field.
2. **Defense in depth**: source-class firewall stops being the single point of failure. Even if a firewall bug lets an extracted value into a control-arg slot, the planner never had the value to compare/decide on.
3. **Generalizable**: the same model can apply to record fields like slack_channel.name — display projection becomes "always handle, with comparison primitives" rather than "depends on field-by-field tuning."
4. **Bug surface narrows**: extract is the choke point. Today, every place an extracted value flows (derive, execute compile, planner reasoning, compose) needs to handle untrusted content. With handles, only extract+derive+compose touch values; planner never does.

## Cons

1. **New primitive surface**: need handle-equality, handle-bucket-membership, and similar comparison primitives the planner can call. Real new architectural surface area.
2. **Derive becomes more central**: any reasoning over extracted content moves to derive worker. More LLM calls (cheap, but architectural shift).
3. **Render-time trust on compose**: the planner can't independently verify that compose rendered the right value from a handle. If compose has a bug or is compromised, the rendered text could be wrong. (Note: this is already the situation for resolved records — not a new attack surface, but worth flagging.)
4. **Refactor scope**: every existing extract consumer (planner reasoning, derive, execute, compose) needs to switch from value access to handle access. Touches a lot of code.
5. **Loss of cross-state textual reasoning**: today the planner can do checks like "the extracted IBAN doesn't match any of the resolved transactions: CH93..., GB29..., SE35..., US122... → so this is suspicious, blocked()." That reasoning would have to move to a tool primitive ("does extracted handle X equal any value in resolved bucket Y?"). The reasoning is the same, but the planner can't articulate it as text — it has to delegate.

## Reasoning chain from the conversation (for re-litigation)

The first-pass argument against handle-only extract output was that it would (a) prevent the planner from cross-checking extracted values against resolved state, (b) push reasoning into derive (more LLM calls), (c) workers still see content anyway, (d) compose has to resolve handles which makes it a content-reader. On re-examination:

(a) WRONG — handle-equality primitives can do the cross-check without exposing values. Architecturally possible; just needs primitive surface area.
(b) FINE — derive LLM calls are cheap; architectural cleanliness is worth it.
(c) IRRELEVANT — workers are inherently in the trust boundary. The planner-cleanness invariant is specifically about the planner, not the workers.
(d) WRONG — compose already reads untrusted content; that's its job. Saying "compose becomes a content-reader" is meaningless because it already is. Compose can't initiate write actions; rendered output is gated by display projection.

The remaining genuine concern is (5) above: the planner loses textual cross-state reasoning. That's real but recoverable via primitives. Net assessment: handle-only extract is probably the right architectural direction.

## Acceptance / decision points before implementing

1. Inventory every place extract output flows. Identify which of those need real value access vs. could work with handles.
2. Specify the comparison primitive surface: which checks does the planner currently do textually that need handle-equivalents? (handle-eq, handle-in-resolved-bucket, handle-as-control-arg-source-class)
3. Decide on handle scope: per-extract-call? per-field? per-worker-session? Which gives the right uniqueness/equality semantics?
4. Confirm compose can render extracted handles without changing the worker prompt structure.
5. Plan the migration: extract worker change, planner prompt addendum, downstream consumers, security tests. Estimate touch surface.
6. Re-run the bench-grind-20 attack matrix on the refactor to confirm no ASR regression and no utility regression.

## Updated framing (2026-05-06)

This ticket re-scoped after follow-up discussion. Key changes:

**1. Scope is broader than extracted scalars.** This is "all worker → planner output as handles" — the architectural ratchet that closes the slack channel-name leak (`slack_channel.name` field visible in planner's view) and the banking IBAN extract-output visibility together. The principle: anything coming back to the planner from a worker should be a handle, not a value. That was the original intent of handles; current implementation half-applies it.

**2. Treat as a separate configuration layer, not a refactor.** Lock in current implementation (display projection + source-class firewall — empirically 0/873 ASR). Build handle-everywhere as a NEW configuration set that runs side-by-side with the current model. This avoids destabilizing what works while letting us measure the stricter model independently. Two tiers:
   - Layer 1 (current): per-field display projection + source-class firewall. Production-ready, locked in.
   - Layer 2 (new): handle-everywhere with worker-delegated comparisons. Stricter "planner sees no untrusted content" invariant. Defense-in-depth.

**3. Ordering/ranking/matching: workers handle it, not framework primitives.** Earlier I argued for new framework primitives (handle-eq, handle-bucket-membership, ranking helpers). On reconsideration: the planner can just delegate to derive. "Rank these channels by activity" → derive sees the values, returns sorted handles. "Find slack_channel where name matches 'general'" → derive does the match. The framework already has the right surface. The only genuinely-new framework primitive worth building is deterministic handle-equality (e.g., "does this extracted handle point to a value already in resolved bucket X?") — narrow, well-defined, deterministic, no LLM call needed.

## What to relitigate before implementing

The earlier reasoning chain had four objections to handle-only output, three of which were wrong on re-examination (see prior note). The remaining real concern was "planner can't textually cross-check values" — discharged by the worker-delegation argument: derive does the cross-check, returns handles + decision. So the design is sound; the question is investment cost vs. defense-in-depth value.

The case for prioritizing this:
- Closes both the slack channel-name leak (32 sessions) and banking extract-visibility (8 sessions) with one architectural change instead of per-field tuning
- Makes the security invariant ("planner sees no untrusted content") structurally enforced rather than relying on display-projection correctness
- Source-class firewall stops being the single point of failure

The case for not yet:
- Layer 1 has 0/873 ASR. Empirically the firewall holds.
- Layer 2 is a real framework refactor + worker prompt updates + bench re-validation. Multi-week effort.
