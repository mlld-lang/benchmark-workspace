# Diagnostic hook layer for rig dispatch boundary inspection

Add an optional rig/diagnostics.mld module with hooks that log key values at each dispatch boundary. Purpose: debug integration failures that don't reproduce in isolated spikes.

Today's UT7 investigation exposed 4 independent bugs stacked across 15 layers. Spikes proved each layer correct in isolation; the integration kept failing because each fix revealed the next bug. We had no way to see what value crossed each boundary in the live session.

The diagnostic layer would use mlld hooks (not inline instrumentation) to log boundary-crossing values:

- after dispatchResolve: handles, count, record_type
- after compileExecuteIntent: policy_intent keys, compiled_args keys, update fields
- after callToolWithOptionalPolicy: typeof result, length, error
- after @policy.build: valid, issues, strippedArgs
- after record coercion: instanceKey, field count, null fields

Each hook writes one JSONL line to a diagnostic file. Shape/type/keys/counts only — not full values. Enough to identify which layer dropped or transformed a value.

Enable via config flag (diagnostics: true) or env var. Off by default. The hooks don't touch dispatch code — they observe after the fact.

This is complementary to c-3edc (logging refactor for production observability). c-3edc replaces lifecycle emission with hooks for the bench analyzer. This ticket adds a debugging-specific layer that wouldn't ship in production — it's the 'printf debugging' equivalent for the rig dispatch stack.


## Notes

**2026-04-23T03:18:50Z** After UT7/m-b71c debugging: this would have helped, but the diagnostic schema should explicitly cover the policy intent boundary. Useful additions: (1) role/scope context: role, exe, phase, operation, because top-level @policy.build and role:planner @policy.build behaved differently; (2) per-arg summaries after compileExecuteIntent: arg, role, source_class, auth_bucket, in_compiled_args, in_policy_intent, is_update_field, policy_value_kind, attestation_count; (3) builder issue details, not just counts: valid, issues[].code/reason/tool/arg/phase, strippedArgs; (4) carrier/materialization fingerprint for intent values: typeof, has_mx, has_data, has_text, is_array, attestation_count, to catch cases where printed JSON is identical but proof metadata was lost by a rebind/merge; (5) dispatch branch decisions: which policy intent arm was used, updateFields.length, updatePairs.length, validPairs.length. Also: don't require purity around hooks if it hides the best probe point. For this class, tiny named boundary diagnostics at compile/build/dispatch are more important than avoiding all touchpoints in dispatch code.
