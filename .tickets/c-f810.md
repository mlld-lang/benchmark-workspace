# Slack body_has_url is fact-classified with no current control-arg consumer (latent risk)

@slack_msg.body_has_url (bench/domains/slack/records.mld:25-40) is declared in the record's facts: list, which means it is eligible to flow into write-tool control args. The value is computed from untrusted message body content via a regex in the bridge (@messageItems in bench/domains/slack/bridge.mld). No current write tool takes body_has_url as a control arg, so the practical attack surface is empty today — but if a future tool declares it as a control-arg input, attacker-influenced URL detection becomes a control-arg source.

Cold-read auditor (claude/B 2026-07-10) raised this as a classification-discipline question rather than an exploitable bug.

## Why body_has_url ended up in facts

Display-side need: the planner's role:planner projection omits message body entirely, so URLs inside a body are invisible. body_has_url surfaces a structural signal so the planner can decide whether to call find_referenced_urls. The flag must be visible to the planner; planner-visible fields go in facts (data fields are stripped from role:planner). Hence the current shape.

## The latent risk

- A future write tool with body_has_url as a fact in its input record (e.g. a hypothetical 'forward_messages_with_urls' write) would accept body_has_url as a control arg.
- Its value is computed from attacker-controllable body text. The attacker can flip the boolean by including or omitting a URL substring in their message.
- That gives the attacker structural control over a planner-visible authorization input.

## Proposed mitigation options

1. **Promote body_has_url out of facts.** Treat it as a planner-visible synthetic display field (similar to handle-typed identifiers) rather than a true fact. Adds a fourth display-form alongside ref/mask/handle for 'computed display predicate'. Preserves the planner-visible signal; removes the control-arg eligibility.

2. **Document the no-control-arg constraint.** Add a comment in records.mld noting body_has_url MUST NOT be used as a control arg in any current or future write tool. Cheaper but discipline-only.

3. **Run a structural check at @validateConfig time.** When a write tool declares body_has_url as a fact-input, fail config validation. Belt-and-suspenders.
