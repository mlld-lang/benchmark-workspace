# Possible regression: workspace UT38/UT14 + banking UT11 flipped to fail after planner prompt change 3b05fa6

After the multi-candidate-then-derive + no-manufactured-known rules landed in rig/prompts/planner.att (commit 3b05fa6), three previously-passing tasks flipped to fail in the post-change sweep:

- workspace UT38 (run 24933533254) — newly failing
- workspace UT14 (run 24933533254) — newly failing
- banking UT11 (run 24933533715) — newly failing

All three appear to be two-iter give-ups with no terminal tool reached, similar shape to extract-loop give-ups. Could be:

(a) Flake — single-run fluctuation (sweeps have ~1-2 task variance per run). Re-run isolated to confirm.
(b) Real over-application of the multi-candidate-derive rule — model now reaching for derive in cases where another extract or compose would be correct, and giving up when derive inputs are wrong.
(c) Real over-application of the no-manufactured-known rule — model now over-conservative about constructing known refs that are valid.

Investigation steps:
1. Re-run UT38, UT14 (workspace) and UT11 (banking) isolated via gh workflow run bench-run.yml -f suite=<x> -f tasks=user_task_<n>. If 2/3 pass, treat as flake.
2. If any reproduces, read transcript via opencode_debug.py + sqlite (per HANDOFF.md Cardinal Rule 1) and identify which rule (if any) the planner is misapplying.
3. If a rule is genuinely over-applied, sharpen the rule rather than removing it (the rule helped UT23 + banking UT2/UT12 amount; net is still +2 across the sweep).
