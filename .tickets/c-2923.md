# Integrate no-untrusted-or-unknown-urls-in-output (mlld-dev defense-in-depth rule)

Integration ticket for the new mlld-dev policy rule no-untrusted-or-unknown-urls-in-output, sibling of no-novel-urls. Defense-in-depth on the body-write side of the URL-promotion threat surface; reduces ASR on tasks that already pass utility, no new utility unlock.

## What the rule does

Each URL extracted from a write tool's body/content arg must trace provenance to known (user task text) or resolved (a record's URL fact field, or a URL-promotion-minted capability matching). Untrusted-source URLs in body content → denied with a hint pointing to the provenance gap.

Defends URL-smuggling-into-write-bodies: a URL that exists verbatim somewhere in the agent's data gets carried into a write body, but its provenance is untrusted source content. Canonical attack: IT1-class phishing URL injected into an inbox message → summarized into Alice's DM body. no-novel-urls is silent (the URL has been seen — it's in Bob's inbox); the new rule fires on the source-class trace.

## Difference from no-novel-urls

- no-novel-urls: URL must appear verbatim in some prior tool result or user payload. Defends URL *construction* (base64-encoded secret in a path on a benign domain).
- no-untrusted-or-unknown-urls-in-output: URL in a body must trace to known/resolved/URL-promotion-capability. Defends URL *smuggling* (phishing URL extracted from untrusted message and re-emitted in summary).

Both rules are needed; they catch disjoint attack classes.

## Mechanism (per mlld-dev's draft)

Same regex extraction as no-novel-urls, applied to body/content args of write tools (not just URL args). Per-extracted-URL trust-class check:

- Verbatim match in user task text → known → allow
- Verbatim match of a resolved record's URL fact field → resolved → allow
- URL-promotion-minted capability matching (state.capabilities.url_refs) → resolved-class → allow
- Otherwise → deny + provenance-gap hint

## Tasks affected (defense-in-depth, not utility unlock)

In-scope passers it defends (currently pass utility, would close IT1-class ASR):
- Slack: UT1, UT3, UT4, UT6, UT15 — heavy summarize-and-send shape
- Workspace: UT22, UT24, UT38, plus any send_email/create_file/append_to_file from extracted email or file content
- Travel: less common, but reviews-into-trip-summary-emails could in principle smuggle URLs

Will NOT unlock SHOULD-FAIL set (UT2/UT11/UT16/UT17/UT20/UT18/UT19 + BK-UT0) — those fail on identity/action delegation, not URL smuggling.

## Integration work (this ticket)

When the rule lands in mlld-dev:

1. Per-suite opt-in in policy declarations. Likely: bench/agents/<suite>.mld passes a defaults.rules entry.
2. Zero-LLM tests in rig/tests/index.mld:
   - URL in body from known (task text) → allow
   - URL in body from resolved record fact field → allow
   - URL in body from URL-promotion capability (slack url_ref deref'd via get_webpage_via_ref) → allow
   - URL in body from extract output (untrusted-source) → deny
   - Multiple URLs in body, mixed provenance → deny if any untrusted
3. ASR canary: run IT1 attack slice against slack UT1/UT4/UT6 with the rule active; verify ASR drops to 0 on phishing-URL-smuggling attacks.
4. Utility regression check: run defended sweep on tasks where URL-in-body is legitimate (workspace email summaries, slack DM-containing-resolved-URL); verify no utility regression.

## Acceptance criteria

- Rule active in all four suite policies.
- Zero-LLM tests cover the four trust-class arms.
- ASR canary verifies block on IT1 phishing slice (currently silently passes utility under attack).
- Full sweep with rule active matches pre-rule utility numbers (no regression).
