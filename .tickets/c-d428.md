# Correlate false positive: both control args from same handle rejected as cross-record. Banking UT2 — model passes id and recipient both from r_scheduled_transaction_7, correlate rejects 5 times. Model eventually succeeds by dropping recipient. Spike needed: reproduce with synthetic data and check if factsources carry matching instanceKey through the session boundary.


## Reopened (2026-04-24, bench-grind-6 session)

I closed this prematurely thinking my @correlatedRefKey JS-unwrap fix
worked. It didn't. UT9 passed during the verify run only because the
planner happened to discover it could omit `recipient` and pass only
`id` — a single-control-arg call bypasses the correlate check entirely.
UT2 and UT12 still fail with the same `cross_record_mixture` error
firing on byte-identical refs to `r_scheduled_transaction_7`.

### What I learned

The dedup in @correlatedControlArgsCheck builds keys via the template
`\`@ref.record:@ref.handle\`` and dedupes via `@unique.includes(@key)`.

- Synthetic test (plain `{record: "...", handle: "..."}` JS object
  literals) — the original code dedupes correctly; .includes() works
  on plain primitive strings.
- Live test (refs that flow through @compileExecuteIntent →
  compileScalarRef → backing_refs construction) — the same code does
  NOT dedupe. Two byte-identical refs produce two distinct keys.

The values produced by the live compile path are wrapper-bearing
StructuredValue strings rather than plain primitives. When interpolated
into a backtick template, the resulting "key" carries wrapper identity
that defeats `.includes()` value-equality. This is the broader bug:
**StructuredValue wrappers persist through field access and template
interpolation**, so any code that uses string-key dedup or value-
equality on fact-derived strings will hit this. The correlate check is
the first surface that exposed it; anything depending on factsources
flowing through session boundaries with plain-string equality will
rediscover it.

### What did NOT fix it

Reaching into JS to manually unwrap `.data` / `.text` is a layering
violation (mlld qs warns against it) and didn't actually solve the
underlying wrapper-persistence issue across the full path. Reverted.

### Repro to keep

Live compile test (preserved here, not in gate so unrelated work can

```mlld
var @correlateLiveIntent = @compileExecuteIntent(

    operation: "share_note_with_contact",

      recipient: { source: "resolved", record: "contact", handle: "r_contact_alice.example", field: "email" }
    }

  "Send to alice.example."
)
let @check = @correlatedControlArgsCheck(@correlateLiveIntent.compiled_entries, ["note_id", "recipient"])
>> Currently produces: { ok: false, error: "correlate_control_args_cross_record_mixture", ... }
>> Should produce: { ok: true }
```

### Suggested fix direction

Use mlld-native unwrapping (e.g. `@stringValue` from runtime.mld) when
building the dedup key:

```mlld
exe @correlatedRefKey(ref) = [
  let @record = @stringValue(@ref.record)
  let @handle = @stringValue(@ref.handle)
  => when [
    !@record.isDefined() || !@handle.isDefined() => null
    * => `@record:@handle`
  ]
]
```

But this is band-aid territory unless we also confirm that template
interpolation of a wrapped string CAN produce a plain primitive (or
that two wrapped values with identical underlying text compare equal
under .includes()). If neither is true, file a parallel runtime ticket
against ~/mlld/mlld for the broader wrapper-persistence-through-template
behavior — many other dedup / equality / cache-key paths in rig will
silently misbehave the same way.
