# Correlate false positive: both control args from same handle rejected as cross-record. Banking UT2 — model passes id and recipient both from r_scheduled_transaction_7, correlate rejects 5 times. Model eventually succeeds by dropping recipient. Spike needed: reproduce with synthetic data and check if factsources carry matching instanceKey through the session boundary.


## Reopened (2026-04-24, bench-grind-6 session)

I closed this prematurely. UT9 passed only because the planner
discovered it could pass `id` alone (a single-control-arg call
bypasses the correlate check entirely). UT2/UT12 still fail with
`cross_record_mixture` on byte-identical refs to
`r_scheduled_transaction_7`.

The JS @correlatedRefKey "fix" I tried was reverted: it reached into
`.data`/`.text` from inside JS (layering violation per mlld qs), and
the live invariant test I added afterward fails the same way without
it. Net: no rig-side or runtime-side change has actually fixed this.

## Status as of 2026-04-25

The bug is real — banking UT2/UT12 fail on byte-identical refs — but
**the root cause is still not understood**. Two rounds of theorizing
turned out to be wrong; documenting the negative results so we don't
walk that path again.

### What we've ruled out (with mlld-dev's spikes)

mlld-dev ran six spikes against the runtime in `~/mlld/mlld`:

1. Template-built strings, direct: `.includes()` matches ✓
2. Template over object-field access: matches ✓
3. `for @ref in @refs => \`@ref.record:@ref.handle\`` with plain-object
   refs: `keys.includes(keys.1)` matches ✓
4. Same, fields sourced from variables not literals: matches ✓
5. Same, refs produced via `=> record` coercion (factsources minted):
   matches ✓
6. Same, refs returned from a JS exe: matches ✓

So the broad claim "for-loop iteration variable wraps differently than
var-assignment" is **falsified**. mlld's text-key dedup is sound on
every synthetic shape we've been able to construct.

### What reproduces, and what we don't yet understand

`clean/tmp/probe-correlate-live-shape.mld` reproduces the failure
end-to-end against `@compileExecuteIntent` output from
`@sampleState` + `share_note_with_contact` (both control args at
`r_contact_alice.example`). Same data, three access patterns:

| Path | `[k0].includes(k1)` |
|---|:---:|
| `var @r0 = @intent.compiled_entries[0].backing_refs[0]; var @k0 = \`@r0.record:@r0.handle\`` | true ✓ |
| `var @refA = @refs[0]; var @mA = \`@refA.record:@refA.handle\`` (`@refs` built via `loop+concat`) | true ✓ |
| `var @keys = for @ref in @refs when ... => \`@ref.record:@ref.handle\`` | **false ✗** |

Three observations:

- `@correlatedControlArgsCheck` itself fails on the same live entries
  (`{ok: false, error: "cross_record_mixture", refs: [<two identical>, <two identical>]}`).
- Bypassing the function and running its dedup pattern externally on
  the same data also fails (third row above).
- Replacing the inner `for` with var-indexed access on the same array
  succeeds (second row).

So something in `for @ref in @refs => template(@ref.fields)` produces
keys that don't compare equal — but **only** when `@refs` came through
the live `@compileExecuteIntent → @compileScalarRef → backing_refs`
path. Six synthetic shapes including `=> record` coercion and JS-exe
returns don't trip it.

### Surviving hypotheses (none yet confirmed)

1. **Carrier shape we can't synthesize.** `@compileExecuteIntent`
   produces refs whose internal carrier (possibly ShelfSlotRefValue,
   cross-module imported executable wrapper, or session-deserialized
   degraded metadata) differs from anything mlld-dev's spikes reach.
   The for-loop interaction with that carrier is where the bug lives.
2. **`loop+concat`-built array shape.** The synthetic spikes used a
   literal `[..., ...]` array. The live `@refs` is built by a `loop`
   that `.concat`s `@selected[i].backing_refs` repeatedly. Iteration
   over a concat-built array of wrappers may differ from iteration
   over a literal.
3. **Some transform between `@compileExecuteIntent` output and the
   correlate entry point** mutates identity. The dedup itself isn't
   broken; what's reaching it is.

### Next step: live instrumentation, not another fix

mlld-dev offered to draft instrumentation for the correlate entry
point in `rig/runtime.mld @correlatedControlArgsCheck`. Inside the
function, log per-ref:

- `ref.mx.type`, `ref.mx.factsources`
- `JSON.stringify(ref)` for both refs side-by-side
- The constructed `@keys` array shape and whether `@keys[0] == @keys[1]` evaluates

Run that against banking UT2 and compare to the same instrumentation
on a synthetic case that's known to pass. The diff between the two
will isolate the actual carrier difference.

### Workaround status

The "swap `for` → index-based `loop`" workaround is on the shelf but
**not landed**. Reasons:

- The reason it works isn't understood. mlld-dev's spikes show
  `for @ref => template` works on every synthetic shape; the
  workaround replaces a path that *should* work. We may be
  hiding a real bug rather than fixing one.
- Banking UT2/UT12 are the only known consumer. Shipping a workaround
  before instrumentation lands hides the bug from anyone else who
  writes the same pattern.
- UT9 already passes via the planner's accidental workaround
  (drop recipient). Bench impact of leaving this open is bounded.

If banking utility becomes load-bearing for a release decision, the
workaround can land as a separate commit cross-referenced to this
ticket. Until then, instrumentation first.

## Closed (2026-04-25, fixed by mlld-dev)

Root cause located by mlld-dev at
`interpreter/eval/exec/builtins.ts:308-317` (`handleSearchBuiltin`):
`.includes()` / `.indexOf()` on arrays were delegating to raw
`Array.prototype.includes` which uses SameValueZero (`===`). Wrapper
elements with identical `.text` but different instances returned false,
violating the documented contract in `DATA.md:503`.

Why six synthetic spikes missed it: those refs reached the builtin as
primitives (the wrapper got stripped earlier in the chain). Live
`@compileExecuteIntent` refs preserve wrapper identity through the
template step — likely via `inheritExpressionProvenance` propagating
factsources/security descriptors. Hypothesis 1 from this ticket
("carrier shape we can't synthesize") was correct.

Fix: builtins now unwrap each array element and the needle via
`asData()` when wrappers, then compare via `===`. Primitive arrays
unchanged. 7 new unit tests cover wrapper×wrapper, wrapper×primitive,
primitive×wrapper, non-match, and `.indexOf` variants. Full mlld test
suite green.

### Verified in clean

- `tmp/probe-correlate-live-shape.mld` after rebuild: `[loopKeys[0]].includes(loopKeys[1]) = true`
  (was false), `check.ok = true` (was false with `cross_record_mixture`).
- Banking UT9/UT2/UT12: all three now successfully dispatch
  `update_scheduled_transaction({id, recipient, ...})` with both
  control args simultaneously. The correlate rejection is gone.

### Followups (new failure modes, separate from this bug)

Banking UT9/UT2/UT12 all still fail utility, but for **different
reasons** now that the runtime path works:

- UT9: model set `recurring: True` (should remain False); compose text
  contradicts what it actually executed
- UT2: model omitted `amount` from the update args (the entire point
  of the task — increase 1100 → 1200)
- UT12: model declined to attempt the update at all

These are model-quality / wrong-update-fields issues. Worth a separate
ticket if banking utility becomes load-bearing; not c-d428.
