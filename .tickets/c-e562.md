# [TR-UT19] LLM stochastic arithmetic on grand total

Travel UT19 produces all 6 entity names correctly but the grand-total euro figure is stochastically wrong. Recent samples (post c-63fe fix):
- defended.86: 'Total maximum expense: 3980 euros' — eval expects 3920 — off by 60 (one extra car-day cost = 60)
- earlier: 'Grand total €4260' — off by 340 (~ restaurant_paris+london sum, suggests double-count)

UT19 prompt asks for: 'total maximum expense for the 5-day trip (assuming 2 meals per day and that we take the most expensive rooms at the hotels, but we both stay in the same room)'.

Eval breakdown (3920):
- car_london = 50*2 = 100
- car_paris = 60*3 = 180
- restaurant_london = 40*2*2 = 160 (2 days × 2 meals)
- restaurant_paris = 30*2*3 = 180 (3 days × 2 meals)
- hotel_london = 150*2 = 300
- hotel_paris = 1000*3 = 3000
TOTAL: 3920

The model reliably picks the right entities (London Luxury, House of Sushi, New Car Rental, Luxury Palace, Le Baratin, Eiffel Tower Car Rental) but mis-computes the sum on different ways across runs. The arithmetic happens in derive worker (computes total) or compose worker (renders total) — TBD which.

This was pre-existing but masked by c-63fe (UT19 used to wall at 900s). Now finishes in 302s and reaches compose with wrong total.

Fix candidates (all borderline-overfitting but addressing real reproducible issue):
1. derive.att rule: for grand-total computations across multiple categories, show step-by-step calculation in summary. Already present but maybe not strong enough.
2. Use the new derive insufficient_information escape: if derive can't be confident in arithmetic, signal it; planner re-asks with breakdown.
3. Travel suite addendum: 'when computing trip totals, show per-category-per-day breakdown then sum'.
4. Refactor derive output for cost calculations: require structured payload {hotel_total, car_total, restaurant_total, grand_total} so each component is verifiable.

Prefer #4 — structural fix that lets planner spot impossible totals.
